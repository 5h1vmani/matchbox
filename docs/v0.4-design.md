# Matchbox v0.4 design

Status: proposed. Date: 2026-05-29. Supersedes nothing in v0.3 that
works; extends it. Read [v0.3-design.md](v0.3-design.md) first — this
document assumes that build as the baseline and only describes the delta.

Code is the source of truth. Where this spec and any older doc disagree,
the spec authoring task already reconciled against the v0.3 code as it
exists on 2026-05-29 (schema at `src/matchbox/core/schema.sql`, contracts
in `schemas/*.v1.json`).

---

## 1. What v0.4 is

v0.3 is a disciplined CV tailor: a tagged bullet library, deterministic
selection, honest assembly. The v0.3 audit found four things that block
the product's stated purpose:

1. **The polish pass can introduce facts the library never verified.**
   `--polish` validates form (voice rules) and keyword presence, not
   truth. It is the one code path through which fiction can reach a CV.
2. **It does not find jobs.** Discovery pulls from company slugs the user
   already named. Scoring is lexical token overlap, not fit.
3. **"ATS-friendly" is an unverified proxy** — a regex over the text of
   the PDF we rendered ourselves. No real parser ever sees it.
4. **BYOK is true in the architecture and false in the surface.** The app
   holds no model (good), but the README and four templates funnel every
   user to Claude Code, and the contract file is named `CLAUDE.md`.

v0.4 fixes the foundations these sit on by changing one thing at the
core:

> **Stop storing a bullet library. Store a verified evidence graph of a
> person, and render every CV, cover letter, and answer as a *projection*
> of that graph. Never as free text.**

That single reframe makes the truth guarantee a property of the type
system instead of a rule we ask a model to obey, and it gives discovery,
fit scoring, and interview prep a real substrate to reason over.

The v0.4 theme in one line: **from a bullet library to a verified
evidence graph; from copy-paste-to-Claude to agent-native; from lexical
scoring to honest fit.**

---

## 2. Principles

Carried forward from v0.3 unchanged:

* The app holds no LLM client. The brain does only the irreducible
  judgment. (ADR-0004 reaffirmed.)
* Selection is deterministic math, not model judgment.
* Local-first. One SQLite DB per profile. No auth, `127.0.0.1` only.
* Versioned JSON-Schema contracts at every app ↔ brain boundary.

New invariants introduced by v0.4:

* **I1 — Projection-only output.** No CV, cover letter, or exported
  artifact may contain text that is not a `rendering` row bound to a
  verified `claim`. There is no code path that writes free prose into a
  document. (Section 5.)
* **I2 — Containment.** A rephrasing of a claim may only assert facts
  present in that claim's structured fields. Any new named entity or
  number not traceable to the claim flags the rendering unapproved and
  routes it back to a human. (Section 5c.)
* **I3 — Verification is owned by the subject.** A claim is verified only
  by the person whose fact it is, recorded in `claim.verified_by`. A
  helper driving the tool for an intern cannot self-verify the intern's
  history. (Section 10.)
* **I4 — Agent-neutral by default.** Nothing user-facing names a specific
  vendor. The instruction file is `AGENTS.md`; `CLAUDE.md` becomes a
  symlink. (Section 7.)
* **I5 — Legibility is measured, not assumed.** The ATS keyword report
  runs over text extracted by a real third-party parser, not over our own
  render. (Section 9.)

## 3. What changes, what we keep

Keep (these were built right):

* `core/`, `discovery/`, the embedding cache, BM25, RRF, MMR, the Typst
  PDF path, the FastAPI + HTMX web shell, the run/handoff loop, the
  pre-commit PII and people-guard hooks, the per-profile DB layout.

Change:

* The library tables (`bullet`, `project`, `summary_variant`) gain a
  truth root above them: the **evidence graph** (`claim`, `evidence`,
  `rendering`). Section 4.
* `assemble.py` selects and renders over claims/renderings. Section 5.
* `polish.py` is retired and replaced by **bounded rephrase** with a
  containment check. Section 5b–5c.
* Scoring gains an **evidence-backed fit report**. Section 8.
* A new **MCP server** wraps the same deterministic functions the CLIs
  call, plus a retained file/CLI contract and an optional built-in-key
  mode. Section 7.
* Assembly emits **`.docx` alongside `.pdf`** and verifies both through a
  real parser. Section 9.
* The web UI gets a **profile switcher**. Section 10.

Remove:

* `apply_polish_to_cv_json()` dead code (`polish.py`, `NotImplementedError`).
* The Claude-specific copy in templates and README.

## 4. The evidence graph (data model)

### 4a. New tables (migration `002_graph.sql`)

Additive, following every v0.3 convention: integer PKs, `_json` columns
guarded by `json_valid()`, ISO-8601 UTC text timestamps, booleans as
`INTEGER 0/1` with `CHECK`, enums as `TEXT` with `CHECK` lists.

```sql
-- The atomic, sourced, verifiable unit of a person's history.
-- Replaces the free-floating text in `bullet` as the root of truth.
CREATE TABLE IF NOT EXISTS claim (
    id             INTEGER PRIMARY KEY,
    experience_id  INTEGER REFERENCES experience(id) ON DELETE CASCADE, -- null for profile-level claims
    kind           TEXT NOT NULL CHECK (kind IN
                       ('accomplishment','responsibility','skill_use','credential')),
    assertion      TEXT NOT NULL,            -- canonical, neutral statement of fact (NOT CV-phrased)
    -- STAR decomposition; filled by elicitation, all optional
    situation      TEXT,
    task           TEXT,
    action         TEXT,
    result         TEXT,
    metric_value   TEXT,                     -- "40%", "12", "$2M" — TEXT keeps units
    metric_kind    TEXT,                     -- "percent_reduction", "team_size", ...
    -- trust gradient (replaces the facts_verified boolean)
    verification   TEXT NOT NULL DEFAULT 'unverified' CHECK (verification IN
                       ('unverified','self_attested','artifact_backed','reference_backed')),
    confidence     REAL,                     -- 0..1, model-estimated extraction confidence
    defensibility  INTEGER CHECK (defensibility IS NULL OR defensibility BETWEEN 0 AND 5),
    needs_detail   INTEGER NOT NULL DEFAULT 0 CHECK (needs_detail IN (0,1)), -- elicitation flag
    source_file    TEXT,
    created_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    verified_at    TEXT,
    verified_by    TEXT                      -- subject identity that confirmed (invariant I3)
);
CREATE INDEX IF NOT EXISTS idx_claim_experience   ON claim(experience_id);
CREATE INDEX IF NOT EXISTS idx_claim_verification ON claim(verification);

-- External corroboration for a claim. A claim with >=1 artifact/reference
-- row can be promoted past 'self_attested'.
CREATE TABLE IF NOT EXISTS evidence (
    id          INTEGER PRIMARY KEY,
    claim_id    INTEGER NOT NULL REFERENCES claim(id) ON DELETE CASCADE,
    kind        TEXT NOT NULL CHECK (kind IN
                    ('artifact_url','document','reference','metric_source')),
    detail      TEXT NOT NULL,               -- URL, file path, reference name+contact
    captured_at TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE INDEX IF NOT EXISTS idx_evidence_claim ON evidence(claim_id);

-- A phrasing of a claim for a target. The ONLY text that may reach a
-- document (invariant I1). Bound to claim_id so it cannot assert a fact
-- outside the claim.
CREATE TABLE IF NOT EXISTS rendering (
    id           INTEGER PRIMARY KEY,
    claim_id     INTEGER NOT NULL REFERENCES claim(id) ON DELETE CASCADE,
    job_id       INTEGER REFERENCES job(id) ON DELETE CASCADE,  -- null = generic/default phrasing
    text         TEXT NOT NULL,              -- CV-ready sentence; voice-rules validated
    carries_json TEXT NOT NULL DEFAULT '[]' CHECK (json_valid(carries_json)),
    approved     INTEGER NOT NULL DEFAULT 0 CHECK (approved IN (0,1)), -- containment gate (I2)
    created_at   TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now'))
);
CREATE INDEX IF NOT EXISTS idx_rendering_claim ON rendering(claim_id);
CREATE INDEX IF NOT EXISTS idx_rendering_job   ON rendering(job_id);
```

`embedding.item_type` gains `'claim'` and `'rendering'`. `item_tag.item_type`
gains `'claim'`. Both are `CHECK`-list widenings in the same migration.

### 4b. Migration of existing data

`002_graph.sql` runs the DDL above, then backfills:

* Each `bullet` becomes one `claim`: `kind='accomplishment'`,
  `assertion = bullet.text`, `experience_id = bullet.experience_id`,
  `source_file = bullet.source_file`,
  `verification = CASE bullet.facts_verified WHEN 1 THEN 'self_attested' ELSE 'unverified' END`,
  `verified_at`/`verified_by` set when it was verified.
* For each backfilled claim, one default `rendering`:
  `text = bullet.text`, `job_id = NULL`, `approved = 1` (existing
  verified text is grandfathered as already-contained).
* `project` rows backfill to `claim` with `kind='credential'` and a
  default rendering likewise.

`bullet` and `project` are kept read-only for one version as a rollback
net, then dropped in `003_drop_legacy_library.sql` once `assemble.py` and
the library UI read exclusively from `claim`/`rendering`. `CURRENT_VERSION`
in `migrations.py` goes to `2`, then `3`.

### 4c. The trust gradient

`facts_verified` was binary. The gradient is ordered and load-bearing:

| verification | meaning | eligible for CV? |
|---|---|---|
| `unverified` | extracted, not yet seen by the subject | no |
| `self_attested` | subject confirmed it is true | yes |
| `artifact_backed` | a URL/doc/PR corroborates it | yes, preferred |
| `reference_backed` | a named reference will vouch | yes, preferred |

The assemble query replaces `WHERE b.facts_verified = 1` with
`WHERE c.verification <> 'unverified'`. Selection (section 8) breaks ties
toward higher trust, so an artifact-backed claim outranks a self-attested
one at equal relevance.

## 5. Projection-only rendering

### 5a. The invariant (I1)

`assemble.py` composes a document from `rendering` rows only. The data
flow is: select claims (the v0.3 IR pipeline, now over claim text) →
resolve each selected claim to a `rendering` for this `job_id` (falling
back to the generic rendering) → lay the chosen rendering texts into the
Typst/`docx` template. The renderer never receives a string that did not
come from a `rendering` row. There is no `--freitext`, no model-authored
paragraph, no cover-letter body typed straight into a file.

This closes audit finding #1 structurally: to put a sentence on a CV you
must first have a claim, and a claim must be verified by its subject.

### 5b. Bounded rephrase (replaces `polish.py`)

The keyword-alignment need is real: a verified claim may not contain the
exact term the JD uses. v0.4 keeps the capability but binds it to a claim.

New contract `schemas/rendering.v1.json` (brain → app, replaces
`polish.v1.json`): a list of `{claim_id, text, carries[]}`. The
deterministic side:

1. validates `text` against `shared/voice-rules.json` (unchanged form
   rules), then
2. runs the **containment check** (5c), then
3. inserts a `rendering` row for `(claim_id, job_id)` with
   `approved` set by the check, then
4. re-renders and re-runs the legibility report (section 9).

A rephrase that fails containment is stored with `approved = 0` and
surfaced in the review-run UI as "needs your confirmation: this wording
adds *Kubernetes*, which is not in the underlying claim." Nothing
unapproved renders.

### 5c. The containment check (I2)

Mechanical, in code, no model:

* Tokenize the claim's `assertion` plus its STAR fields plus
  `metric_value` into a fact-token set (reuse `core/text.py`).
* Tokenize the proposed rendering `text`.
* Compute the rendering's content tokens minus a stoplist minus the
  claim's fact tokens minus the JD's own keyword/variant set for this
  requirement (mirroring a JD term the subject genuinely matches is
  allowed).
* Any **named entity** (capitalized non-sentence-initial token, or a
  known tech term from the tag vocabulary) or any **number** remaining in
  the difference → `approved = 0`.

This is deliberately conservative: it cannot prove truth, but it can
prove that a rephrase introduced no new proper noun or quantity, which is
where fabrication actually lives. The residual judgment ("is this still
the same fact") is the human's at confirm time. v0.3 had no such gate at
all.

## 6. Elicitation (active information gathering)

"Get as much information on a person" cannot be satisfied by parsing an
old CV. The old CV is the floor. v0.4 adds a structured interview, driven
by the brain through the existing handoff, costing the app almost nothing
because the app only defines the schema and surfaces the gaps.

* After ingest, any claim with weak structure (no metric, no STAR
  action/result, low extraction confidence) lands `needs_detail = 1`.
* A new prompt, `elicit my gaps`, drives the brain to interview the user
  on exactly those claims: "You wrote 'led the migration.' How many
  engineers? Before and after numbers? What broke?" Answers update the
  claim's STAR fields and metric, and clear `needs_detail`.
* The `/review` screen shows a "needs detail" filter so a human can see
  what is thin and run elicitation until it is empty.

Deep elicitation — pulling consented artifacts (GitHub contributions,
Scholar, a personal site) to auto-promote claims to `artifact_backed` —
is specified in section 17 as v0.5; the `evidence` table is built now so
that work is additive.

## 7. Agent-native interface (BYOK done right)

The architecture is already model-free. v0.4 makes the *surface* match,
and removes the copy-paste handoff for agents that can do better.

### 7a. MCP server (`matchbox.mcp`)

A new module exposes the same deterministic functions the CLIs already
call as Model Context Protocol tools, so any MCP-capable agent drives
Matchbox natively with no pasted prompts:

| tool | wraps | notes |
|---|---|---|
| `profiles.list` / `profiles.switch` | `core/db` | section 10 |
| `ingest.submit` | `onboarding.ingest_cli` | replaces "ingest my files" |
| `claims.list` / `claims.get` | `core/library` | read graph |
| `claims.verify` | `core/library` | records `verified_by` (I3) |
| `jobs.add` / `jobs.list` | `web/routes/inbox` logic | add by paste/URL |
| `requirements.save` | `jobreqs` | unchanged contract |
| `fit.report` | section 8 | returns fit-report.v1 |
| `assemble` | `assemble` | renders pdf + docx |
| `rendering.propose` | section 5b | bounded rephrase |
| `coverage.get` | `matching/coverage` | parser-verified |
| `run.status.update` | status.v2 | replaces file-watch for MCP agents |

The server holds no model and no key. It is a typed adapter over
deterministic code. It is started by the human once
(`matchbox-mcp` or via the web app's "Connect your agent" panel, which
prints the stdio/SSE config to paste into the agent's MCP settings).

### 7b. File + CLI contract (retained)

The v0.3 file handoff (`work-queue.json` in, `status.json` out, CLIs in
between) stays, verbatim except for the schema bumps below. Agents that
cannot speak MCP — including a human running commands by hand — lose
nothing. MCP is an acceleration, not a requirement.

### 7c. Optional built-in-key mode (off by default)

For a user who has an API key but does not run an agent at all, a small
opt-in adapter performs the four judgment steps (parse, extract
requirements, elicit, rephrase) by calling the user's chosen provider.
It lives behind a `setting` and an explicit env var, ships with no
default provider, and is clearly labelled as the one mode where Matchbox
talks to an LLM. BYOK therefore means *bring your agent or bring your
key*. The deterministic core is untouched by this; it is one more caller
of the same CLIs.

### 7d. `AGENTS.md` (I4)

`CLAUDE.md` is renamed `AGENTS.md` and made vendor-neutral
("your coding agent" not "Claude Code"). `CLAUDE.md` becomes a symlink so
Claude Code still auto-loads it. README and the four templates
(`onboarding/index`, `inbox/_run_started`, `review_run/index`,
`onboarding/review`) lose every "Claude Code" string in favour of "your
AI agent," with Claude Code named once as *an example*.

## 8. Honest fit (scoring + fit report)

The five-dimension lexical rubric (`shared/rubric.json`) stays as a cheap
first-pass triage score. v0.4 adds a second, evidence-backed pass that
runs only for jobs the user is considering, and that is allowed to say
"do not bother."

For a given job, using machinery that already exists (requirement
extraction, embeddings, coverage):

* For each must-have requirement, find the best-matching **verified**
  claim. A requirement is **covered** if that claim's similarity clears
  the floor *and* the claim is verified; **partially covered** if a
  related but weaker claim exists; a **gap** otherwise.
* Derive a seniority/scope signal from experience dates and claim metrics
  (team-size and scope metric_kinds), so an over- or under-qualified
  match is detectable — the thing pure token overlap cannot see.
* Emit `schemas/fit-report.v1.json` (brain or app, deterministic core):
  `overall` 0–10, `recommendation` in
  `{strong_fit, stretch, weak, skip}`, per-requirement coverage with the
  supporting `claim_id`, a `gaps[]` list, and `interview_risks[]` — load-
  bearing claims whose `defensibility` is low (you will be grilled on
  these).

This serves the real purpose: spend the user's scarce time on the right
jobs, and walk into the interview knowing where the pressure will come
from. The fit report is shown on `/inbox` before a run is started and is
the input a thoughtful user triages on.

> Note: this reuses the embedding floor already tuned in
> `tests/test_fastembed_integration.py`. No new model.

## 9. Verified legibility (the ATS claim, made real)

### 9a. `.docx` alongside `.pdf`

Many ATS pipelines parse `.docx` more reliably than PDF. Assembly emits
both from the same `cv.json`: the Typst path for the PDF (unchanged) and
a new `docx` renderer built from the structured `cv.json` (python-docx,
single column, standard headings — the same ATS-conservative layout the
Typst template already follows per v0.3 section 13b). Output:
`runs/<id>/output/<job>/cv.pdf` and `cv.docx`.

### 9b. Real-parser check + test corpus (I5)

Replace "regex over our own PDF text" with "regex over text a real parser
extracted." Add a vendored open-source resume parser as a dev/runtime
dependency; run it over both `cv.pdf` and `cv.docx`; assert that name,
email, section headers, and each must-have keyword survive extraction.
The keyword-presence report (`matching/coverage.py`) now consumes the
parser's output, not `pypdf` over our own render.

Pin it: a `tests/corpus/` of `(jd, claims) -> expected coverage` cases,
asserted in CI. This is the empirical check the v0.3 "ATS-friendly" claim
never had. We still do not claim to "bypass" anything; we claim, and now
test, that a standard parser reads what we wrote.

Typst remains an external binary. v0.4 adds a clear preflight in the web
UI ("PDF rendering needs Typst — install it / we bundled it") and falls
back to `.docx`-only output if Typst is absent, so a non-technical user
is never left with a crash. (Audit: v0.3 raised `FileNotFoundError`.)

## 10. Multi-profile, first-class

Isolation already works (`people/<slug>/matchbox.db`, `MATCHBOX_PROFILE`).
v0.4 adds the UX the constraint requires (help friends, family, interns,
students):

* A **profile switcher** in the web shell. Active profile is session-
  scoped (single-user localhost, so a module-level "current profile" set
  by the switcher and read by `web/deps.py` is sufficient; no auth).
* **Create / list / switch** profiles from the UI. Each new profile lazily
  migrates its own DB on first request, exactly as today.
* **Helper workflow (I3):** the person driving may be a helper, but
  `claims.verify` records `verified_by`, and the UI states plainly that
  the subject must confirm their own facts. A profile is a portable
  folder the subject can take with them.

No multi-user, no auth, no cloud sync (carried non-goal). This is profile
switching on one trusted machine, nothing more.

## 11. Outcome tracking (measurement, not learning-to-rank)

The `application` table already records `status`
(draft → applied → interview → offer → rejected → withdrawn),
`response_type`, and `response_at`. v0.4 surfaces it: an **Insights** view
showing response rate by fit-band and by keyword-coverage, so the user
can see whether the tool is working.

> **ALERT — revisiting a prior decision.** v0.3 design section 16 and the
> 2026-05-14 roadmap explicitly **rejected outcome-based
> learning-to-rank** and **auto-apply** as permanent non-goals. This spec
> proposes outcome *measurement and human-readable insight only*. It does
> **not** propose feeding outcomes back into the ranker automatically, and
> it does **not** propose auto-apply. Both remain rejected. If you want
> the system to *learn* from outcomes later, that is a deliberate reversal
> to make with eyes open, not something this spec sneaks in. Flagging
> because it sits one short step from the proposed Insights view.

## 12. Contracts (new and changed schemas)

All bump or add under `schemas/`. `schema_version` mismatch stays a hard
error everywhere (unchanged behaviour).

* **`ingest.v2.json`** — replaces `bullets` with `claims` carrying
  `assertion`, optional STAR fields, `metric_{value,kind}`, `evidence[]`,
  and extraction `confidence`. `schema_version` const `2`.
  `ingest.v1.json` is retained; `ingest_cli` accepts both and upconverts
  v1 (a v1 bullet becomes a `self_attested`-eligible unverified claim).
* **`rendering.v1.json`** — replaces `polish.v1.json`. `{schema_version,
  run_id, job_id, renderings:[{claim_id, text, carries[]}]}`.
* **`fit-report.v1.json`** — new. `{schema_version, job_id, overall,
  recommendation, requirements:[{text, status, claim_id}], gaps[],
  interview_risks[]}`.
* **`status.v2.json`** — adds optional `fit` and `legibility` blocks per
  job (overall fit, parser-verified keyword hits/misses). v1 still
  accepted by the watcher.
* **`work-queue.v1.json`** — unchanged.
* **MCP** — the tool list in 7a is the contract; each tool's args/results
  reuse the JSON schemas above so MCP and file paths stay isomorphic.

## 13. Tech stack and module changes

* New deps: an MCP server lib; `python-docx`; a vendored resume parser.
  All additive, all local, no network at runtime except the optional
  built-in-key mode.
* New modules: `src/matchbox/graph/` (claim/evidence/rendering CRUD +
  containment check), `src/matchbox/mcp/` (the server), `src/matchbox/
  legibility/` (docx render + parser check). `polish.py` deleted.
* `assemble.py` shrinks: selection and projection split into
  `matching/select.py` (claims) and a thin `graph/project.py`.
* `core/library.py` gains claim/evidence/rendering functions; remains the
  only place SQL touches the library.

Determinism stays in the app; judgment stays in the brain. The MCP server
does not change that line — it is a transport, not a brain.

## 14. Build plan (sequenced)

Milestones in the v0.3 house style. M0–M5 are v0.4. M6+ are listed in
section 17 as v0.5 and are designed-for now but not built.

* **M0 — Evidence graph.** `002_graph.sql` + backfill + `graph/` CRUD +
  embedding/item_tag widening. Tests: backfill preserves every v0.3 row;
  trust gradient maps correctly. No UI change yet.
* **M1 — Projection-only assembly.** `assemble.py` reads claims/renderings;
  invariant I1 enforced; smoke test asserts no document text exists
  without a backing rendering. Drop `bullet`/`project` reads.
  `003_drop_legacy_library.sql`.
* **M2 — Bounded rephrase.** Delete `polish.py`; add `rendering.v1.json`,
  the containment check (I2), and the "needs confirmation" review-run UI.
  Tests: a rephrase adding a new proper noun or number is rejected; an
  in-claim rephrase passes.
* **M3 — Agent-native.** `mcp/` server; `AGENTS.md` rename + symlink;
  strip vendor copy from README and four templates; "Connect your agent"
  panel. Built-in-key mode behind a setting (thin, optional).
* **M4 — Honest fit.** `fit-report.v1.json`; evidence-backed coverage +
  seniority signal; fit report on `/inbox`; `status.v2.json`.
* **M5 — Verified legibility.** `docx` renderer; vendored parser; coverage
  reads parser output; `tests/corpus/`; Typst-absent graceful fallback.

Each milestone ships its tests. M1's smoke test is the v0.4 equivalent of
the v0.3 blank-page guard: it must fail if any rendered line lacks a
backing rendering row.

Elicitation (section 6) is small and rides on M2's review UI plus an
`AGENTS.md` prompt; include it in M2 if cheap, else first thing in v0.5.

## 17. Open questions / explicitly v0.5+

Designed-for now (tables/columns exist), built later:

* **Deep elicitation via consented artifacts** (GitHub/Scholar →
  `artifact_backed`). The `evidence` table is ready.
* **Adversarial review.** A "skeptical hiring manager" pass that flags
  weak or generic renderings before output. Pure brain work over the
  graph; no schema change.
* **Discovery that searches.** Graph-driven cross-board recall so the
  user need not name companies. Larger; its own design doc.
* **Interview-prep projection.** The fourth projection of the graph
  ("claims on your CV → questions you will get"). Falls out of section 8's
  `interview_risks` plus the STAR fields.
* **Local-model brain** (Ollama) as a built-in-key provider.
* Open question: should `rendering` cache per-job phrasings across runs,

## 18. Non-goals (carried and new)

Carried from v0.3 / the 2026-05-14 roadmap, unchanged: no auto-apply, no
multi-user, no auth, no cloud sync, no public profiles, no paid tier on
the OSS, no automated learning-to-rank.

New for v0.4: no model in the deterministic core (the built-in-key mode
is a separate, opt-in caller); no "beat the ATS" trickery (we optimise
legibility and truthful keyword coverage, nothing adversarial); no free-
text generation into any document (invariant I1).
