# Matchbox v0.3 — Design Document

> Status: draft for build. Supersedes v0.2. Target audience: the person building
> this (in Claude Code) and the maintainer.

## 1. What we are building

A **local desktop app** that turns a person's scattered career material into a
**tagged content library**, scans real job-board APIs for matching roles, scores
them deterministically, and — with **Claude Code as the reasoning engine** —
assembles tailored CVs and cover letters by *selecting and lightly polishing*
pre-written components. The user reviews the output, downloads it, and applies to
each job themselves.

The app never auto-submits applications. The app never embeds an LLM. All judgment
work is done by Claude Code through a file-based handoff.

## 2. Principles

1. **Deterministic plumbing, LLM for judgment only.** Scanning, scoring,
   component selection, assembly, and rendering are fixed, tested code. The LLM
   is used only where judgment is irreducible: parsing messy input, extracting
   JD requirements, polishing wording, drafting cover letters.
2. **The app holds state and does I/O. The brain does judgment.** They communicate
   through a versioned file contract, never a live coupling.
3. **Assemble, don't rewrite.** Each achievement is written once, tagged richly,
   and reused. Variation between CVs comes from *selection and ordering*, not
   regeneration. This is what makes the tool fast and cheap.
4. **Never invent facts.** The LLM may select, reorder, and lightly rephrase
   user-provided facts. It may not fabricate experience. Gaps are flagged, not
   filled.
5. **Local-first.** All data stays on the user's machine. The only thing that
   leaves is JD text and the user's own profile text, sent by Claude Code to
   whichever model the user configured there.
6. **ATS-safe output.** Every template and palette must parse cleanly in
   applicant tracking systems.
7. **Lean.** One template, curated choices, no feature museum. No command palette,
   no keyboard-shortcut sprawl, no premature abstraction.

### Engineering invariants

Non-negotiable for the build; they follow from the principles above and from the
failure modes the v0.2 audit surfaced.

* **One source of truth.** The SQLite DB is authoritative. The `runs/` folder is
  derived and transient — never edited back as truth. `requirements_json` and the
  `embedding` table are *caches*: each carries version info (`model_version`, a
  JD hash) so staleness is detectable and a recompute is clean.
* **Schemas defined once.** The handoff contracts live as JSON Schema files in
  `schemas/`; the app and the brain both validate against them. No schema is
  written out twice.
* **Truthfulness is enforced in code, not by prompt.** The `facts_verified`
  eligibility filter in `matching/` is the authoritative gate. The `CLAUDE.md`
  rules are guidance; the deterministic filter is enforcement. Never rely on the
  LLM to honour an invariant the code can guarantee.
* **Fail loud.** A failure never looks like an empty success — a failed ATS
  probe is distinct from "no new jobs", a parse error surfaces, nothing is
  swallowed. This was v0.2's most damaging habit.
* **One responsibility per module.** `assemble.py` is a thin orchestrator:
  selection lives in `matching/`, rendering in a render module, the keyword
  check in its own. Web route handlers stay thin.
* **DRY applies to knowledge, not coincidence.** Deduplicate rules, constants,
  and schemas. Do not merge two functions that merely look alike but encode
  different decisions — that is the over-abstraction trap v0.2 fell into.
* **Least privilege.** The app holds no LLM credentials and binds to localhost
  only. The brain touches only `runs/`, the DB, and the named scripts. PDF
  serving is sandboxed to the run output folder. Uploaded files are untrusted
  input — parsed with maintained libraries, never executed.
* **Test the critical path.** The matching layer is deterministic — unit-test it
  on fixed fixtures. The end-to-end smoke test (M5) is mandatory.

## 3. Architecture

Two halves joined by a handoff folder.

```text
┌──────────────────────────────┐         ┌──────────────────────────────┐
│  THE APP  (deterministic)    │         │  THE BRAIN  (Claude Code)    │
│                              │         │                              │
│  - Onboarding UI             │         │  - Parses uploaded files     │
│  - SQLite: profile, library, │  files  │  - Selects components per JD │
│    jobs, runs                │ <─────> │  - Polishes (within rules)   │
│  - ATS pollers               │  in a   │  - Drafts cover letters      │
│  - Deterministic scoring     │ shared  │  - Runs assemble.py          │
│  - Job inbox UI              │ folder  │  - Writes status.json        │
│  - assemble.py + Typst       │         │                              │
│  - File watcher              │         │  Reads CLAUDE.md for rules   │
└──────────────────────────────┘         └──────────────────────────────┘
            no LLM code                       no app state of its own
```

End-to-end data flow:

```text
1. Onboarding   user drops files ─► app stages them ─► Claude Code "ingest"
                ─► library populated ─► user reviews & confirms in app
2. Discovery    app polls ATS APIs ─► jobs into SQLite
3. Scoring      deterministic rubric ─► score + breakdown into SQLite
4. Triage       user selects jobs, toggles CV / cover per job in the UI
                ─► app writes runs/<id>/work-queue.json
5. Tailoring    Claude Code reads the queue ─► per job: select components,
                polish, cover letter ─► runs assemble.py ─► writes PDFs +
                status.json
6. Review       app watches status.json ─► shows results, previews, gaps,
                and the apply link per job
7. Apply        user clicks the apply link, applies manually, marks "applied"
```

Why Claude Code as the brain: the app needs zero API-key handling, zero prompt
code, zero retry/cost logic. The agent is more capable than a single API call —
it can read multiple files, run scripts, and check its own work. The interface
is just files, so the brain is **pluggable**: a future "in-app key" mode can
write the same files without a terminal. Do not build that mode for v1.

## 4. Data model (SQLite, no YAML)

YAML is dropped entirely. It only ever earned its place as a hand-editing format;
with file-drop onboarding and UI editing, it earns nothing and contributes a
whole class of parse errors. One SQLite DB per profile at
`people/<name>/matchbox.db`. JSON is used only for the handoff files.

```text
profile          (id, full_name, email, phone, location, links_json, headline)
target           (id, role_families_json, dream_companies_json,
                  locations_json, comp_json, exclusions_json)

experience       (id, company, role, start_date, end_date, location, sort_order)
bullet           (id, experience_id, text, has_metric, facts_verified,
                  source_file, created_at)
project          (id, name, text, url, facts_verified)
skill            (id, name, category, proficiency)
summary_variant  (id, label, text)          -- alternative positioning summaries

tag              (id, facet, value)         -- see taxonomy in section 5
item_tag         (item_type, item_id, tag_id)  -- links a tag to any component
embedding        (item_type, item_id, model_version, vector)  -- cached vectors
                                                -- for components and requirements

ats_source       (id, ats_type, slug, company, country, sector, enabled)
job              (id, source, company, title, location, url, apply_url,
                  jd_text, requirements_json, posted_at, fetched_at, status,
                  score, score_breakdown_json)

run              (id, created_at, status)
run_job          (run_id, job_id, want_cv, want_cover, palette, font)
application      (id, job_id, run_id, cv_path, cover_path, status,
                  applied_at, response_type, response_at)

setting          (key, value)               -- chosen palette, font, etc.
```

Notes: `item_tag` is polymorphic (`item_type` ∈ bullet | project | skill |
summary_variant) — acceptable for a small single-user DB; if it gets awkward,
split into per-type join tables. `facts_verified` defaults to `false` on import
and flips to `true` only when the user confirms a component during onboarding
review. Job `status` ∈ new | scored | selected | tailored | applied | skipped |
rejected. `job.requirements_json` caches the structured requirements the brain
extracts from the JD (section 5b). `embedding.vector` is a stored blob; the
`embedding` table covers both library components and cached requirement vectors,
keyed by `model_version` so a model change can trigger a clean recompute.

## 5. Component matching — taxonomy + retrieval

This is the part that decides whether CVs come out sharp or generic. A *taxonomy*
is not made "SOTA" — it is made small, interpretable, and good at hard filtering.
The intelligence belongs in the *matching layer*. The design keeps a slim
faceted taxonomy for hard constraints and routing, and uses hybrid retrieval
(dense + sparse) plus diversity-aware selection for soft relevance. Every
technique here is proven, runs locally, and requires no model training.

### 5a. The slim taxonomy (hard filters + routing only)

Embeddings carry thematic nuance, so the manual taxonomy shrinks to what
machines genuinely need as discrete handles:

| Facet | Type | Used for |
|---|---|---|
| `role_family` | categorical | routing — eligibility + which `summary_variant` |
| `tech` | keyword set | hard requirement matching (exact skill hits) |
| `seniority` | enum (light) | filter — keep junior bullets out of senior CVs |
| `impact` | enum (light) | selection prior — metric > scope > recognition |

`has_metric` stays a boolean column. The old `theme` facet is **dropped** as a
required field — embeddings cover thematic similarity far better than a
hand-maintained list. `role_family` values are seeded from the user's stated
targets, so the taxonomy is personal, not global. The brain auto-suggests all of
these at ingest; the user only confirms.

**Authoring rule (write once).** Each real achievement is written exactly once as
a `bullet`. CVs differ because *different bullets are selected and ordered
differently*, never because the same achievement was regenerated.

### 5b. The matching pipeline

```text
JD text
  │
  ├─[brain] extract requirements ─► [{type: must-have|responsibility|nice,
  │                                    text, keywords}]   (saved to the job)
  │
  ▼ (deterministic, in assemble.py)
embed each requirement                    embed each library component
  │                                         (cached; recomputed on edit)
  ▼
per (component, requirement) score:
  dense   = cosine(embed_component, embed_requirement)
  sparse  = BM25(component_text, requirement_keywords)
  taghit  = exact `tech` overlap bonus
  │
  ▼
fuse dense + sparse rankings with Reciprocal Rank Fusion (RRF, k=60)
  → one calibration-free relevance number per (component, requirement)
  │
  ▼
apply priors:  × recency weight  × has_metric boost
  │
  ▼
MMR greedy selection under constraints (max 4 bullets/role, one-page budget):
  repeatedly add the component with the highest
     marginal relevance to still-uncovered requirements
     − λ · redundancy with the already-selected set
  │
  ▼
selected component set  +  coverage report
```

Why each piece:

* **Hybrid dense + sparse.** Pure embedding similarity misses hard keyword
  requirements ("must have Kubernetes"); pure keyword search misses paraphrase
  ("built ML infra" vs "machine-learning platforms"). Combining both is the
  standard retrieval pattern. Sparse = BM25 (`rank-bm25`, or ~30 lines).
* **Match against extracted requirements, not the whole JD.** A JD is long and
  noisy; embedding it whole dilutes the signal. Decomposing it into discrete
  requirements sharpens every comparison and makes coverage measurable.
* **RRF for fusion.** Reciprocal Rank Fusion combines the dense and sparse
  rankings using *ranks*, not raw scores — no normalization, no per-model
  threshold tuning. Robust when there is almost no data to calibrate on.
* **MMR for selection.** Picking the top-K bullets independently yields a CV
  where eight bullets make the same point. Maximal Marginal Relevance greedily
  trades relevance against redundancy, so the CV *covers* the JD's distinct
  requirements. It is ~15 lines, deterministic, and — because coverage is
  submodular — greedy selection carries a provable (1 − 1/e) quality bound.

### 5c. Two judges — the matcher and the ATS

The matching engine above and the applicant tracking system that ultimately
reads the CV judge differently, and the design must serve both:

* **The matcher (this tool)** is *semantic*. Embeddings understand that "built
  ML infrastructure" and "machine-learning platforms" are the same thing. It
  correctly finds and selects a relevant bullet even when its wording differs
  from the JD — that is the defence against missing real experience over a
  vocabulary mismatch.
* **A real ATS** is mostly *literal*. It parses the CV into fields, stores it,
  and lets recruiters keyword-search it; any match score it shows is keyword
  overlap with light stemming and, sometimes, a synonym list. Assume it is not
  semantic — no given ATS can be relied on to be smart.

State plainly to avoid over-engineering for a myth: the "ATS auto-rejects most
resumes" claim is largely marketing by resume services. Most ATS do not
auto-reject on a content score. The real failure modes are narrower — a CV the
parser mangles (see section 13b), or a CV that never surfaces in a keyword
search.

The consequence: selecting the right bullet is necessary but not sufficient. If
the selected bullet does not literally contain the JD's must-have terms, a
keyword search still misses it. So the assembler reports **two** metrics:

1. **Semantic coverage** — for each must-have requirement, is it genuinely
   supported by a selected component (similarity above a floor)? The honest
   answer to "am I actually a fit?" Uncovered must-haves populate `gaps`.
2. **ATS keyword presence** — a deterministic, literal check: does the final
   rendered CV text contain each must-have term and its accepted variants? This
   predicts what a literal ATS will see. Missing terms are shown to the user
   alongside the bullet that should carry them.

Showing both is the honest position: the tool reports whether the person is a
real fit *and* whether the document will survive a keyword filter — and never
conflates the two.

### 5d. Division of labour

Selection is deterministic math in `assemble.py`. The brain does **not** pick
component IDs — that would put LLM variance in the wrong place. The brain does
only genuine judgment: extracting requirements from a messy JD, and optional
wording polish. Selection stays reproducible, testable, free, and fast.

**Keyword-alignment pass (default for must-haves, not optional cosmetic).**
Selecting the right bullet does not help if its wording misses the JD's term.
After selection, the brain rephrases selected bullets so they carry the JD's
must-have vocabulary — *but only where the new wording is a truthful description
of the same fact*. "Built data pipelines" may become "built ETL data pipelines"
if the work was ETL; it may not gain "ETL" otherwise. This is what a careful
human tailor does. It runs by default for must-have terms the keyword-presence
check (section 5c) flags as missing, and stays bounded by
`shared/voice-rules.json`.

**Truthfulness.** Only `facts_verified = true` components are eligible. A keyword
is never injected for an experience the user does not have. If the JD demands
something the library lacks, it surfaces as an uncovered requirement in `gaps` —
never fabricated.

### 5e. Local + lean

The library is small (hundreds of components). Use a lightweight embedder that
avoids the heavy PyTorch stack — an ONNX model via `fastembed`, or static
embeddings — benchmarked during the build for the size/quality trade-off.
Vectors are stored as blobs in SQLite (`embedding` table); the `sqlite-vec`
extension is optional and only worth it well beyond v1 scale.

### 5f. Explicitly NOT doing this (over-engineering traps)

* **Outcome-based learning-to-rank.** Tuning selection on which CVs got
  interviews sounds SOTA; it is a trap. One user produces a handful of labels,
  the outcome signal is dominated by factors unrelated to the CV, and any model
  overfits noise. A light heuristic — boost components the user keeps and edits —
  is the most that is justified.
* **DPP** for diversity — MMR delivers the same intent at a fraction of the cost.
* **Fine-tuning / SPLADE / ColBERT** — far beyond a few-hundred-item library.
* **Unsupervised theme discovery** (clustering bullets) — too little data per
  user to be stable.
* A **cross-encoder reranker** over the candidate set is the one legitimate
  future upgrade if quality needs another notch — noted for v2, not built in v1.

## 6. The app ↔ brain handoff contract

All exchange happens in `runs/<run-id>/`. Both schemas are versioned; a
mismatched `schema_version` is a hard error on both sides.

**`work-queue.json`** — written by the app when the user confirms a triage run:

```json
{
  "schema_version": 1,
  "run_id": "2026-05-21-001",
  "created_at": "2026-05-21T10:00:00Z",
  "profile_db": "people/shiva/matchbox.db",
  "jobs": [
    {
      "job_id": 42,
      "company": "Modal",
      "title": "Forward Deployed Engineer",
      "jd_text": "...full JD text...",
      "apply_url": "https://...",
      "want_cv": true,
      "want_cover": false,
      "palette": "slate",
      "font": "source-serif"
    }
  ]
}
```

**`status.json`** — written by the brain, updated as it works:

```json
{
  "schema_version": 1,
  "run_id": "2026-05-21-001",
  "status": "running",
  "jobs": [
    {
      "job_id": 42,
      "cv_status": "done",
      "cover_status": "skipped",
      "cv_path": "runs/2026-05-21-001/output/42/cv.pdf",
      "cover_path": null,
      "gaps": ["JD asks for Terraform; no verified component covers it"],
      "notes": "Selected 11 bullets across 3 roles."
    }
  ]
}
```

Output folder per job: `runs/<run-id>/output/<job-id>/` containing `cv.pdf`,
`cover.pdf` (if requested), `cv.json` (the assembled structure, for re-render),
and `changes.md` (a plain-language diff vs the master CV, for the review screen).

The app file-watches `status.json` and re-renders the review screen on change.

## 7. CLAUDE.md (draft)

This lives at the repo root. The brain reads it on every run. Draft:

```markdown
# Matchbox — instructions for the reasoning engine

You are the brain for Matchbox. The app has prepared work for you.

## Your job
Process the newest run in `runs/`. For each job in `work-queue.json`:
1. Read the JD. Extract its requirements as a typed list (must-have /
   responsibility / nice-to-have), each with text and keywords. Save them:
   `python -m matchbox.jobreqs save --job <job_id> --file <reqs.json>`
2. Run the assembler — it embeds, scores, and selects components
   deterministically, then renders the CV:
   `python -m matchbox.assemble --run <run_id> --job <job_id>`
3. Read the coverage report it returns. Uncovered must-haves are expected and
   are recorded as `gaps`. Do not cover them by inventing content.
4. If `want_cover` is true, draft a cover letter from the selected components
   (listed in the assembler output) and the JD, following
   `shared/voice-rules.json`, then render it:
   `python -m matchbox.assemble --run <run_id> --job <job_id> --cover`
5. Optional: polish the wording of selected bullets to mirror JD keywords —
   rewording only, bounded by the voice rules.
6. Update `status.json` for that job.

## Hard rules
- NEVER invent experience, employers, dates, metrics, or skills. Only
  facts_verified components are eligible, and the assembler enforces this.
- Do NOT pick or hand-rank components yourself — `matchbox.assemble` does
  selection deterministically. Your judgment goes into requirement extraction
  and optional polish, nowhere else.
- Do NOT assemble or render PDFs yourself. Always call `matchbox.assemble`.
- If the JD needs something the library lacks, leave it as an uncovered
  requirement — never fill a gap with fiction.
- Polishing is rewording only, bounded by `shared/voice-rules.json`.
- Stop and report if `schema_version` does not match.

## Onboarding mode
If asked to "ingest", read every file in `inbox/`, extract experiences,
bullets, projects, and skills, tag them per the taxonomy, and write them to the
DB with facts_verified = false. The user will confirm them in the app.
```

## 8. Tech stack + module breakdown

One language for everything written by hand: **Python (3.12+)**. The retrieval
and embedding ecosystem is Python, the salvaged v0.2 code is Python, and there
is no hot path that would justify a second systems language — polling is
I/O-bound, the DB is tiny, and the embedding model runs native ONNX code
regardless of host language. Rust would buy nothing measurable here.

* **Logic, app, CLIs, matching** — Python. SQLite via stdlib `sqlite3`, HTTP via
  `httpx`, embeddings via `fastembed` (ONNX, no PyTorch), BM25 via `rank-bm25`;
  RRF and MMR are ~50 lines of plain Python.
* **Web UI** — FastAPI + Jinja templates + HTMX + Tailwind. Effectively zero
  JavaScript: HTMX drives interactivity over HTML, with a little Alpine.js only
  where genuine client state is needed. No React, no Node build step, no npm.
* **CV templates** — Typst markup (`.typ` files), a small declarative language.
  Typst itself is a prebuilt Rust binary called as a subprocess — consumed, not
  written.
* **Handoff + config** — JSON (the work-queue / status contracts) and Markdown
  (`CLAUDE.md`). The brain is Claude Code, not code in this repo.
* **Packaging** — PyInstaller for a single launch-the-server binary, or a thin
  `pywebview` window. Both are pure Python. Tauri is an option but its only gain
  is a native window at the cost of a Rust shell — defer it.

Net: you write Python, a little Typst, and HTML templates. Rust is only ever
consumed as prebuilt binaries.

```text
src/matchbox/
  core/         SQLite access, models, migrations
  onboarding/   file staging into inbox/, library review helpers
  discovery/    ATS pollers (salvaged + expanded), add-company logic
  scoring/      deterministic job rubric
  matching/     embeddings, BM25, RRF fusion, MMR selection, coverage
  jobreqs.py    CLI: save the brain's extracted JD requirements
  assemble.py   CLI: deterministic selection (via matching/) + Typst render
  web/          FastAPI + HTMX UI, file watcher
  templates/    Typst CV/cover templates + the app's HTML templates
```

The app runs subprocesses (Typst) but contains no LLM client. `jobreqs.py` and
`assemble.py` are deterministic CLIs the brain invokes.

## 9. Onboarding flow

1. User opens the app, lands on an empty-state onboarding screen.
2. Drag-and-drop zone accepts many files: old CVs (PDF/DOCX), LinkedIn data
   export, free-text `.md` notes, plus a paste box for typing rough stories.
   Message: "the more you give me, the better." Files are staged into `inbox/`.
3. User runs Claude Code: "ingest my files" (the app shows a ready-to-paste
   prompt so the terminal step is a single paste).
4. The brain parses everything, extracts and tags components, writes them with
   `facts_verified = false`.
5. **Review screen (non-negotiable).** The user sees every extracted component
   grouped by experience, can edit text, fix tags, delete noise, and confirm.
   Confirming flips `facts_verified` to true. Parsing errors compound downstream;
   this screen is the guardrail.
6. A short guided form captures targets: role families, dream companies,
   locations, comp range. (Culture fit deferred to v2.)
7. The component library is a permanent, editable screen — the user's owned
   "warehouse," not a hidden table.

## 10. Discovery

Salvage the v0.2 pollers — Greenhouse, Lever, Ashby — they hit clean public JSON
APIs and are the strongest code in the old repo. Then expand.

Candidate additional sources (exact endpoints to be verified during the build,
not assumed): **Workable, SmartRecruiters, Recruitee, Teamtailor, Personio,
Breezy HR, JazzHR.** Workday is per-tenant and messier — treat as a stretch.

Two hard requirements the old repo missed:

* **Users add their own companies** via the UI (ATS type + slug), stored in
  `ats_source`. Without this, a first scan returns nothing for most users.
* **Failures are visible.** A failed probe must read differently from "no new
  jobs." Add a retry with backoff and surface per-source scan status.

Scanning is on-demand (a button) for v1; a scheduler is a later add.

## 11. Scoring

Keep the deterministic rubric. Remove the dishonest placeholders from v0.2
(`comp_score` and `cultural_score` were hardcoded 3.0). v1 dimensions, all
computed from real data:

* role-title match (JD title vs target role families / positive filters)
* skills overlap (JD text vs the user's `tech` tags and bullets)
* company tier (dream-company list)
* location / remote fit
* red flags / exclusions

Show the breakdown in the UI. Never display a number that is partly a constant.

## 12. Assembly

`assemble.py` is deterministic and unit-tested. Input: run id, job id, palette,
font — it reads the job's cached requirements and the component library from the
DB; it is never handed a pre-picked component list. Steps: embed requirements
and components (cached), run the hybrid scoring and MMR selection in `matching/`
(section 5b), enforce the one-page budget, build `cv.json`, run `typst compile`
with the template plus palette/font inputs, and emit `cv.pdf`, `changes.md`, and
a coverage report (semantic coverage + ATS keyword presence, section 5c — the
latter computed on the text extracted back out of the rendered PDF). Cover
letters: the brain drafts the text, `assemble.py --cover` renders it with the
cover template.

Because selection lives here and not in the LLM, the whole path is reproducible
and unit-testable — the same job and library always yield the same CV. The app
can also re-run `assemble.py` itself when the user changes palette or font after
the fact; that path needs no brain.

## 13. Design system (lean)

Two separate systems. Do not mix them.

### 13a. App UI

Do not invent a design system — constrain Tailwind. One token sheet, referenced
everywhere, so every screen the build produces looks the same:

* **Semantic colours:** `bg`, `surface`, `border`, `text`, `text-muted`,
  `accent`, plus `success` / `warning` / `danger`. One accent colour, chosen
  once.
* **Type scale:** one UI font (system stack, or Inter), five sizes
  (xs/sm/base/lg/xl), three weights, defined line-heights.
* **Spacing:** Tailwind's 4-pt scale, but use only `1,2,3,4,6,8,12` — no others.
* **Radius / shadow:** one radius, one shadow. No more.

### 13b. CV output

One Typst template. The user chooses a **font** and a **predefined palette** —
not a free colour picker (which lets users build unreadable, ATS-hostile CVs).

Each palette is four roles: **page** (white or near-white only), **body**
(near-black), **heading**, **accent/rule**. Starter set, all to be contrast- and
ATS-checked with `design:accessibility-review` before shipping:

| Palette | Page | Body | Heading | Accent |
|---|---|---|---|---|
| `slate` | #FFFFFF | #1A1A1A | #1A2B3C | #3C6E9C |
| `ink` | #FFFFFF | #161616 | #000000 | #555555 |
| `forest` | #FFFFFF | #1A1A1A | #1E3A2F | #2F6B4F |
| `claret` | #FFFFFF | #1A1A1A | #3A1E26 | #8C3A4F |
| `bronze` | #FDFDFB | #1F1B16 | #2E2417 | #8A6A38 |

ATS-safety — *colour* rules baked into every palette: page background is always
white or near-white, body text is always near-black, no text is ever placed on a
coloured fill or inside an image, colour is used only for headings and thin
rules.

ATS-safety — *parsing* rules, a property of the template, not the palette, and
equally important: a CV with perfect keywords still fails if the parser mangles
the layout. The single template must be single-column; use standard section
headings ("Experience", "Skills", "Education"); emit real selectable text only
(never text inside an image); use no tables or text boxes for layout; put nothing
meaningful in the page header or footer (many parsers drop those); use a standard
bullet character; and embed a common font. The template is validated against
this checklist as part of M5, and the keyword-presence check (section 5c) runs on
the extracted text of the *rendered PDF*, so a parsing regression is caught
automatically.

Font pairings (curated, 3–4 options): one serif body (e.g. Source Serif), one
sans body (e.g. Inter / Source Sans), optionally one humanist option. The v0.2
repo already bundles **Atkinson Hyperlegible** — keep it as the accessible
choice.

### 13c. Voice

Two short guides, not a brand book.

* **App UX copy** (~half a page): calm, plain, second person, no jargon; buttons
  are verbs ("Scan jobs", "Confirm components"); errors say what happened and
  what to do next. Build with `design:ux-copy`.
* **CV / cover-letter voice** (`shared/voice-rules.json`): port v0.2's
  `voice-rules.yaml` — banned words, banned openers, no em-dashes, no
  contractions, metrics encouraged. This file is also the polish-pass guardrail.

What we are **not** building: logo system, marketing brand, illustration style,
motion system. Premature for v1.

## 14. Build plan (sequenced v1)

Each milestone ends with a working, demonstrable slice and its own test.

* **M0 — Repo migration.** Archive v0.2, branch `v0.3`, scaffold the new
  `src/matchbox` tree.
* **M1 — Data layer + library screen.** SQLite schema, models, and a UI to add
  and tag components manually. Proves the warehouse before any LLM.
* **M2 — Onboarding ingest.** File-drop staging, the brain's ingest mode, the
  review/confirm screen.
* **M3 — Discovery.** Salvage Greenhouse/Lever/Ashby, add 2–3 new ATS types,
  add-company UI, visible scan status.
* **M4 — Scoring + triage UI.** Honest rubric, job inbox, the select/flag
  screen, `work-queue.json` writer.
* **M5 — Matching + assembly + template.** The `matching/` module (embeddings,
  BM25, RRF, MMR, semantic coverage + ATS keyword-presence check), `jobreqs.py`,
  `assemble.py`, ONE working Typst template validated ATS-parse-safe (section
  13b), the palettes. Unit-test the selection on fixed inputs. **Includes an
  end-to-end smoke test that asserts a non-empty, well-formed PDF whose extracted
  text contains the expected keywords** — the single check whose absence let v0.2
  ship a blank page.
* **M6 — Handoff loop.** `CLAUDE.md`, `status.json` writer/watcher, the review
  screen with previews and apply links. First true end-to-end run.
* **M7 — Cover letters + polish.** Cover-letter drafting, the diff view, font
  and palette switching, empty states.

## 16. Open questions / explicitly v2+

* Culture-fit scoring (hard to compute honestly — deferred).
* Local-model brain (Ollama) as an alternative to Claude Code.
* In-app bring-your-own-key mode (no terminal) — the handoff being file-based
  keeps this possible without a rebuild.
* Multiple template layouts (one parameterized template ships first).
* Cross-encoder reranker over the matching candidate set — the one sanctioned
  quality upgrade if v1 selection proves not sharp enough (section 5f).
* Scheduled background scanning.
* Auto-apply — explicitly out of scope, permanently, for quality and risk
  reasons.
* Outcome-based learning-to-rank — explicitly rejected, not deferred: too few
  labels per user and too noisy a signal to learn from (section 5f).
