## 2. Architecture: two intelligence providers, one honesty gate

This is the answer to "can we have both manual handoff and BYOK?" — yes, and they are complementary, not redundant.

### 2.1 Manual handoff — the workhorse (default, no key)

* **Who:** the user's own Claude Code session (a coding agent with filesystem + DB access).
* **Trigger:** the app writes typed intents to the `agent_task` queue and creates a run (`scoring.runs.create_run` → `runs/<id>/work-queue.json`, schema-validated). The SPA surfaces a copyable "process run X" prompt. The user pastes it into Claude Code.
* **Owns:** requirement extraction (`extract_reqs` → `jobreqs save`), the tailoring run (`assemble` — deterministic component selection + render), prep briefs, follow-up / thank-you drafts.
* **Grounding:** the verified-fact library (`facts_verified` gate). Voice-gated on the polish path (`polish.validate_voice`). Schema-validated at each CLI boundary.
* **Output:** artifacts (`cv.pdf`, `cover.txt`, `coverage.json`, `changes.md`) + `runs/<id>/status.json`, which the app file-watches.
* **Honesty:** strong by construction — deterministic selection, provenance from fact ids, gaps left empty.

### 2.2 BYOK — the optional live layer (additive, needs the user's key)

* **Who:** the user's provider key (Anthropic / OpenAI) via `MBAI.stream`, routed through a **localhost FastAPI streaming proxy** (`POST /api/ai/stream`). Not browser-direct: that is CORS-fragile, and the proxy holds nothing, just terminates SSE locally. The key never leaves the device.
* **Owns:** interactive, in-the-moment prose where waiting on a manual handoff is too slow — "Regenerate" a cover paragraph, tailor a screening answer, redraft a follow-up. Streamed live into the panel.
* **Honesty guardrails (these are what let BYOK coexist under decision #3):**
  1. **Grounded in real facts, not inlined constants.** The browser client is fed verified facts via `GET /api/library/facts?verified=1`, never the prototype's hardcoded strings. Same §13 system prompt.
  2. **Honest about what is and isn't enforceable.** `polish.validate_voice` guards **form** — banned words, banned openers, em-dashes, contractions, word-count. It does **not** verify factual grounding, and a fabricated metric or employer passes it clean. So the voice gate is a *style* check, not a fabrication guard. Do not claim otherwise. BYOK prose runs through `POST /api/voice-check` for **form/voice consistency only** (with prose-scoped word limits — see §4); factual safety comes from the three real levers below, not the gate.
  3. **The real anti-fabrication levers (since free prose cannot be deterministically fact-checked):** (a) the BYOK client is fed **only verified facts** as source material via `GET /api/library/facts?verified=1`, plus the §13 "never invent" system prompt; (b) **the user always reads and sends** — nothing auto-submits, so the human is the final fact check; (c) **anything fact-bearing and high-trust stays on the deterministic manual-handoff path** (CV bullet selection is provenance-traced by construction); BYOK is for rewording and voice, not for asserting new facts.
  4. **Provenance at the honest altitude.** Free prose cannot get per-fact pills cheaply, so the document is labeled "drafted from your library" / "your voice"; per-paragraph fact pills appear only where a real link exists. No fabricated `f1`/`f2` labels.
  5. **Key isolation.** Per-profile key moves out of plaintext `localStorage` to a `0600` secret file under `people/<slug>/` (read only by the localhost proxy), bound to the `mb_profile` cookie / active profile. Fixes the prototype's `window.__mbProfile='maya'` vs backend-slug mismatch.
* **Degradation:** no key → AISource badge says "Demo — add a key," streams the canned fallback, and the **real** grounded artifact still comes from the manual handoff. Honest by construction.

### 2.3 Why they do not conflict

They write to the same artifact store and pass the same voice gate, but they never own the same task at the same time. Manual handoff owns **runs** (heavy, repo-grounded). BYOK owns **interactive prose regeneration** (light, live). One honesty contract, two producers. "Regenerate" is an explicit user action that replaces the artifact body and re-runs the voice-check; the AISource badge shows which producer last touched it. No silent overwrite.

```text
                 ┌─────────────────────────────┐
   user pastes   │   Claude Code (manual)      │  runs: extract_reqs, tailor,
   "process run" │   filesystem + DB access    │  prep, drafts  → artifacts + status.json
                 └──────────────┬──────────────┘
                                │            ┌──────────────────────────┐
   FastAPI (localhost) ◄────────┴────────────┤  artifact store + voice  │  single source of truth
        │  GET /api/library/facts            │  gate (polish.validate)  │  one honesty contract
        │  POST /api/voice-check             └──────────────────────────┘
        │  POST /api/ai/stream  ◄── per-profile key (0600)
        ▼
   Browser MBAI.stream (BYOK)  →  live: regenerate cover para, tailor answer, redraft note
        └── no key → demo fallback (badged), real artifact still from manual handoff
```

## 5. Phased plan

Sequenced for sanity even though scope is the full journey. Each phase ends green (the repo's "green suite before next phase" discipline).

### Phase A — Make the honest core visible (low effort, high trust)

* Wire the serialize stubs: read `coverage.json` into the Role/Tailoring view-model; serialize real `salary`; add the `role_family` tagger so role-scoped pay is honest.
* Add `partial` band + evidence-fact-id to `coverage.json` (small lifts from existing similarity matrix).
* Port the ⌘K palette: `designs/v1.2/Palette.jsx` → `frontend/src/ui/Palette.tsx`; add `commands` + `Cmd/Ctrl+K` handler to `Shell.tsx`; port palette CSS.
* Global copy pass #1: correct the "nothing leaves this machine" line to honest-but-warm wording (your key, your provider, we store nothing).
* **Acceptance:** fit / eligibility / coverage / salary all real in the SPA; palette navigates; no "computed" claim shown that isn't.

### Phase B — BYOK live layer + the gate as code

* Build `GET /api/library/facts`, `POST /api/voice-check`, `POST /api/ai/stream`.
* Per-profile secret store + Settings wiring; bind key to `mb_profile`; **wire or remove** the non-functional "local"/Ollama provider option (no dishonest affordances).
* Port `ai.js` / `aiui.jsx` / `Settings.jsx` into `frontend/`; point `MBAI.stream` at `/api/ai/stream`; honest AISource badge.
* Wire `MBQueue` around live streams (real for BYOK generations only). The seeded ambient demo progression stays clearly demo — it does not pretend the app is working in the background (manual handoff = no background work).
* **Acceptance:** cover-letter regenerate streams live with a key, demo without; BYOK output voice-checked before save; key per profile and outside the data file.

### Phase C — Application packet (the design's biggest addition)

* **Résumé tab:** wire existing tailoring artifacts (`cv.pdf`, `coverage.json`, `changes.md`). Mostly real.
* **Cover tab:** `assemble --cover` (manual) + BYOK regenerate (live); per-paragraph pills only where a real fact link exists; voice gate (form only) on both producers; factual safety comes from fact-grounded prompting + the user reading and sending, not from the gate.
* **Questions tab:** answer-library matcher (depends on Phase D's table — sequence accordingly, or land the table here); each answer shows its source; optional questions stay `needs-you`/blank. **Note:** ingesting the target form's literal screening questions has no honest source — questions are pasted/entered, not scraped. Flagged in UI.
* **Submit tab:** reuse `create_application`; create at stage `applied` with `applied_at`; +7d follow-up via `default_action_for('applied')` (change 5→7). **Label it "reminder," not "scheduled task"** — there is no scheduler; it is a due-date computed on read. Warm-path nudge **cut** (People cut).
* **Acceptance:** 4-tab packet works end to end; submit creates a tracked application at `applied` with a follow-up reminder.

### Phase D — Answer library + Library Answers tab

* Migration 008 `answer` table; extend `schemas/ingest.v1.json` + `ingest_cli` for Q&A; reuse `/review` confirm flow; `used_count` incremented at selection; `/api/answers` router; port Library Answers tab.
* Per-company tailoring: agent_task `draft_answer` kind (manual) **or** BYOK live — both voice-gated, both recombine only what the user wrote.
* **Acceptance:** answers carry verified/unverified status + usage counts; tailoring never fabricates company claims (gate-enforced).

### Phase E — Interview loop + debrief (honesty-trimmed)

* Migration 009 `interview_round` + `debrief`; DAL; serializer to `Round[]`.
* Port Workspace timeline; one-tap debrief capture (real write — recruiter/HM/onsite/values, status, focus).
* prep `agent_task` payload carries the prior debrief so the manual handoff sharpens the next prep — **labeled an assisted draft, not computed intelligence.**
* **Cut/trim:** "auto-calibration from debrief sentiment" and "auto-surface library gaps" are shown **side-by-side** (your debriefs next to outcomes), not asserted as computed statistics. Rounds are **manual entry** — no calendar/ATS/email sync exists, so no auto-populated timeline.
* **Acceptance:** rounds + debriefs persist; next-prep carries prior debrief; no n-of-1 self-report dressed as a calibrated number.

### Phase F — Momentum coach + rejection learning (honesty-trimmed)

* `momentum(conn, week_start, target)` pure fn next to `insights/metrics.py` + `/api/insights/momentum`: real weekly pace (applied / interviews / follow-ups) + healthy/push/rest threshold. The "tell the user to rest when healthy" behavior is a threshold on a real count.
* Structured `close_reason` capture at close (extend `log_response`/`set_stage`); deterministic GROUP BY for categories; uncaptured closures stay "unknown."
* **Trim:** benchmark figures ("typical 20-40%") are labeled generic guidance, not the user's measured benchmark. Narrative insight/nudge text never asserts an unrecorded cause.
* **Acceptance:** momentum is real arithmetic; rejection categories derive only from captured reasons; no fabricated causes, no warm-path nudge (People cut).

### Phase G — Offers (honesty-trimmed)

* Wire `offers/benchmark.py` (add `range` + `basis` text) + insights; port Offers screen.
* Weighted comparison: persist weights/scores **or** keep client-side — be explicit which; do not show it as a backend computation if it is client-side.
* **Negotiation:** status tracking only. No fabricated counter-offer strategy. The `counter` artifact is a user-written draft, voice-gated.
* **Acceptance:** benchmark honest (own-pool, "none" on empty); negotiation surfaces status, never invented strategy.

### Phase H — First-run / onboarding + final honesty + a11y

* Build the "drop a CV → review facts" flow (port Intake) over the existing upload + `ingest_cli` + `/review`. The extraction is the manual handoff (the user runs "ingest my files"); the **intake UI tells the truth** that a model parses the files (per the corrected privacy copy), it does not claim on-device magic.
* Trust model: **verified / unverified (binary)** — no trust-ring % or verified/thin/gap shown as computed until real corroboration exists.
* Final copy pass; accessibility audit (drawers already focus-trapped; palette keyboard-first; verify hover-only actions also fire on focus).
* **Acceptance:** a new person can go from zero to a reviewed fact library; no un-backed "computed" claim ships; a11y pass scheduled/done.

## 8. Risks & sequencing notes

* **Manual handoff vs full journey tension (accepted):** the Apply packet, ambient chip, and arrival toasts ship demo-badged or BYOK-live, not app-driven. If the team later wants the real ambient experience, the AgentRunner (`POST /api/runs` + SSE + subprocess) is a clean fast-follow — the queue and artifact spine already exist, so it is additive, not a rewrite.
* **Phase C depends on Phase D's `answer` table** for the Questions tab. Either land migration 008 at the start of C, or run D before C.
* **Embeddings dependency (hard prerequisite, not graceful):** `assemble_one` unconditionally constructs `FastEmbedEmbedder()` and calls `cached_encode` — there is **no keyword-only / BM25 fallback at the assemble layer**. Without the model, assemble **crashes, it does not degrade.** The `bge-small` model (~30 MB) is **not shipped**; `fastembed` downloads it at first use. Either (a) treat the model download as a hard install-time prerequisite and bundle/pre-warm it, or (b) build the BM25-only fallback the prototype copy implies. Do not claim graceful degradation until (b) exists.
* **`role_family` tagger** is the one non-trivial Phase A item; without it, salary role-scoping silently matches zero rows.
* **Effort:** sized as A/B = small, C/D/E = medium, F/G = small-medium, H = medium. No day estimates given without team size — would be fabrication.
