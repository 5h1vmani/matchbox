# Matchbox v0.3 — Execution Plan

Source of truth: `docs/v0.3-design.md`. This file is the *how I will build it*,
not a re-statement of the design. Read alongside, not instead.

## Working agreement

* Branch: `v0.3` off `main`. All work lands there. Merge to `main` only when M6
  passes the end-to-end smoke test.
* Commit cadence: per logical unit inside each milestone. Each commit compiles
  and (where it has tests) passes them.
* Delivery: M0 through M7 in one continuous run, no pause between milestones.
* If I hit an irreducible decision (e.g. an ATS endpoint I cannot verify
  without an account), I stop and ask. I do not guess.

## Cross-cutting decisions made up-front

These are decided once here so they do not need re-deciding per milestone.

1. **Embedder.** `fastembed` with `BAAI/bge-small-en-v1.5` (384-dim, ONNX,
   ~30 MB). Static embeddings as a fallback only if benchmarking shows the model
   download is a UX problem.
2. **BM25.** `rank-bm25` package. Tokenization: lowercase, strip punctuation,
   keep ASCII letters and digits. No stemming in v1 (BM25 plus dense covers it).
3. **Fusion.** RRF with `k=60`.
4. **MMR.** `lambda=0.5` default, exposed as a setting later if needed.
5. **PDF text extraction (for ATS keyword check).** `pypdf` — pure Python,
   already mature, no native deps.
6. **DOCX parsing (onboarding).** Out of scope for the deterministic side. The
   brain reads files in `inbox/` and writes structured rows. The app only stages
   files.
7. **PDF parsing (onboarding).** Same as DOCX: brain reads, app stages.
8. **Typst.** Detect with `shutil.which("typst")`; if missing, surface a clear
   error pointing at install. Do not vendor the binary in v1.
9. **Web server bind.** `127.0.0.1` only. Never `0.0.0.0`. Default port 8765.
10. **Schemas.** JSON Schema 2020-12 in `schemas/`. Validated with
    `jsonschema` (add to deps in M0).
11. **YAML removal.** v0.2's `shared/rubric.yaml` and `shared/voice-rules.yaml`
    are converted to `shared/rubric.json` and `shared/voice-rules.json` during
    M4 and M5 respectively, and the `ruamel.yaml` dependency is dropped.
12. **Database location.** `people/<slug>/matchbox.db`. Slug is the directory
    name. The app picks the active profile from a `setting` row or env var
    `MATCHBOX_PROFILE`.

## Milestones

### M0 — Repo migration

Goal: a clean `v0.3` branch where the new tree exists and v0.2 is archived,
without losing any salvageable code.

Steps:

1. Create branch `v0.3`. Move uncommitted v0.3 design / audit / plan docs onto
   it first (they should not stay on `main`).
2. `git mv src/matchbox archive/v0.2/matchbox`. Same for `plans/` and any
   stale v0.2-specific docs under `docs/` (decisions stay; v0.2 plans move).
3. Scaffold the new tree:

   ```text
   src/matchbox/{core,onboarding,discovery,scoring,matching,web,templates}/
   src/matchbox/{__init__.py,cli.py,jobreqs.py,assemble.py}
   schemas/{work-queue.v1.json,status.v1.json,job-requirements.v1.json}
   runs/.gitkeep
   ```

4. Update `pyproject.toml` deps: add `fastembed`, `rank-bm25`, `jsonschema`,
   `watchfiles`, `pypdf`, `alpinejs` is a CDN asset (no Python dep). Drop
   `ruamel.yaml` and `anthropic` (the app holds no LLM client).
5. Wire `matchbox = "matchbox.cli:app"` and `matchbox-web = "matchbox.web.app:run"`
   to thin placeholders that print "not implemented in M0".
6. Update `.gitignore` for `runs/`, `inbox/`, `people/*/matchbox.db`.

Done when: `pip install -e .[dev]` succeeds on a clean venv; `matchbox --help`
runs; old tests under `tests/` either move to `archive/v0.2/` or stay only if
they target code that survives the salvage (most will move).

### M1 — Data layer + library screen

Goal: prove the warehouse before any LLM. User can manually add a bullet, tag
it, see it in the library list, edit it, delete it.

Steps:

1. `core/db.py` — single SQLite connection helper. WAL mode. Foreign keys on.
2. `core/schema.sql` — DDL exactly as section 4 of the design. One file, one
   source of truth.
3. `core/migrations.py` — minimal: read `schema.sql`, apply if `migration`
   table empty. Designed so a future v0.4 can add `0002_*.sql` files.
4. `core/models.py` — dataclasses (not pydantic for internal rows; pydantic
   only at the schema boundary). Polymorphic `item_tag` helper functions.
5. `web/app.py` — FastAPI app, Jinja templates, HTMX. Bind `127.0.0.1:8765`.
6. `web/routes/library.py` — list / add / edit / delete bullet, project,
   skill, summary_variant. Tag chips (add/remove inline).
7. `templates/_base.html.j2`, `templates/library/*.html.j2` — Tailwind via
   CDN in M1 (PostCSS build deferred until M7 if needed).
8. Unit tests: round-trip a bullet, attach tags, query by tag, delete.

Done when: starting `matchbox-web`, opening `http://127.0.0.1:8765/library`,
adding three bullets with tags, restarting, all three are still there.

### M2 — Onboarding ingest

Goal: file-drop → brain extracts → user confirms.

Steps:

1. `web/routes/onboarding.py` — empty-state landing page that detects "no
   profile yet"; drag-and-drop into `inbox/`; paste box that saves to
   `inbox/notes-<timestamp>.md`.
2. UI surfaces the ready-to-paste prompt: "Run Claude Code in this directory
   and tell it: *ingest my files*."
3. `CLAUDE.md` at repo root (draft per section 7 of the design). Add only the
   onboarding section in M2; tailoring sections land in M6.
4. `onboarding/ingest_cli.py` — a small CLI the brain calls *after* it has
   parsed files, taking a JSON file with rows to insert. Keeps the file-parse
   judgment in the brain; the deterministic side just writes to the DB. New
   rows land with `facts_verified = false`.
5. `web/routes/review.py` — review screen, grouped by experience. Inline edit
   text, fix tags, delete, "confirm all in this experience" button → flips
   `facts_verified = true` for selected rows.
6. Targets form: role_families, dream_companies, locations, comp, exclusions.
   Saves to `target` row.

Done when: a fixture inbox plus a fake brain invocation populates the library
with `facts_verified = false` rows; the review screen lets me edit and confirm
them; the targets form persists.

### M3 — Discovery

Goal: scan ATS APIs for real jobs and land them in `job`.

Steps:

1. Salvage Greenhouse, Lever, Ashby pollers from `archive/v0.2/matchbox/
   discovery/` into `src/matchbox/discovery/`. Convert any YAML config to a
   `setting`/`ats_source` row. Tighten error handling.
2. Add Workable, SmartRecruiters, Recruitee — endpoints verified at build
   time via a one-off probe script. If an endpoint cannot be verified, the
   poller is not shipped in v1 (logged as deferred).
3. `web/routes/sources.py` — add-company UI: choose ATS type, paste slug,
   probe once before saving. Show per-source last-scan status (timestamp,
   ok / error message).
4. `discovery/runner.py` — single scan-all entry point, called by a Scan
   button. Retries with exponential backoff (3 tries: 1s, 4s, 16s).
5. Failures are visible: a failed source surfaces as a red row in the
   sources list, not a silent zero in the inbox.

Done when: I can add a real Greenhouse slug and a real Lever slug, click
Scan, see jobs land in the DB with `status = "new"`, and a deliberately bad
slug surfaces as an error in the UI.

### M4 — Scoring + triage UI

Goal: honest deterministic score per job and a triage screen to pick which to
tailor.

Steps:

1. Convert `shared/rubric.yaml` to `shared/rubric.json`. Trim the dishonest
   dimensions (`comp_score`, `cultural_score`).
2. `scoring/rubric.py` — five real dimensions from section 11 of the design:
   role-title match, skills overlap, company tier, location/remote fit, red
   flags / exclusions. Each returns a number and a breakdown line.
3. `scoring/run.py` — score every job with `status = "new"`, flip to
   `"scored"`.
4. `web/routes/inbox.py` — triage screen: sortable job list with score and
   breakdown popover; per-row toggles for "want CV" and "want cover";
   palette + font picker shared at the run level. "Start tailoring" writes
   `runs/<run-id>/work-queue.json` (validated against the schema) and flips
   selected jobs to `"selected"`.
5. Schema: `schemas/work-queue.v1.json`.

Done when: scoring a batch of real jobs produces a sensible ranking with
zero hardcoded constants; I can flip toggles, hit Start, and see a
`work-queue.json` written that passes schema validation.

### M5 — Matching + assembly + template

The substance of the build. End of this milestone, an `assemble.py` call from
the terminal produces a real PDF.

Steps:

1. `matching/embed.py` — `fastembed` wrapper. Caches vectors in the
   `embedding` table keyed by `model_version` and a content hash. Recompute
   path is clean.
2. `matching/bm25.py` — thin wrapper over `rank-bm25`. Tokenizer per the
   decision above.
3. `matching/rrf.py` — RRF fusion, `k=60`. Pure function.
4. `matching/mmr.py` — MMR selection under a one-page budget and "max 4
   bullets per role". Returns the selected set plus the (component,
   requirement) coverage matrix.
5. `matching/coverage.py` — two metrics from section 5c: semantic coverage
   (similarity-floor check per must-have) and ATS keyword presence (literal
   check, accepts a small variant list per term, runs on the extracted
   text of the *rendered PDF*).
6. `jobreqs.py` — CLI: `save --job <id> --file <reqs.json>`. Validates
   against `schemas/job-requirements.v1.json`. Writes to
   `job.requirements_json` plus a `model_version` tag.
7. `assemble.py` — orchestrator only: load job + library, embed, score,
   select, render. Reads `runs/<id>/work-queue.json`; writes `cv.json`,
   runs `typst compile`, runs the keyword-presence check on the extracted
   PDF text, writes `changes.md` and a coverage report.
8. `templates/cv.typ` — single template. Single-column, standard headings,
   standard bullet character, no tables/text-boxes, embeds the chosen font,
   palette via `--input`. Validated against the ATS-parsing checklist in
   section 13b before the milestone closes.
9. Palettes: ship all five (`slate`, `ink`, `forest`, `claret`, `bronze`).
   Fonts: ship Source Serif, Inter, Source Sans, Atkinson Hyperlegible
   (already in `shared/fonts/`).
10. Voice rules: convert `shared/voice-rules.yaml` to
    `shared/voice-rules.json`.
11. **Tests.** Unit tests for RRF, MMR, embed cache, BM25 tokenizer,
    coverage. **End-to-end smoke test**: fixture library + fixture JD →
    `assemble.py` produces a non-empty PDF whose extracted text contains
    the expected must-have terms. This single test is the one whose absence
    let v0.2 ship blank pages — it is mandatory.

Done when: the smoke test passes in CI.

### M6 — Handoff loop

Goal: first true end-to-end run. App writes the queue, brain processes it,
app shows results, user clicks Apply.

Steps:

1. Finish `CLAUDE.md` with the tailoring instructions per section 7 of
   the design.
2. Schema: `schemas/status.v1.json`. Both sides validate.
3. `web/watcher.py` — `watchfiles` over `runs/<active>/status.json`. When it
   changes, an HTMX endpoint re-renders the review row.
4. `web/routes/review_run.py` — review screen: per-job card with status, CV
   preview (PDF embed served from the sandboxed run folder), gaps,
   apply link, "Mark applied" button (writes the `application` row).
5. PDF serving: a FastAPI route bound to `runs/<run-id>/output/<job-id>/`
   with path-traversal guards. No file outside that directory is reachable.

Done when: I drop the test JD, run Claude Code, watch the review screen
update in real time, see the PDF preview, and click the apply link.

### M7 — Cover letters + polish

Goal: cover letters, the diff view, palette / font switching after the fact,
empty states.

Steps:

1. `assemble.py --cover` — renders cover letters from text the brain wrote.
   New `templates/cover.typ`.
2. `changes.md` rendered into a "what's different" panel on the review
   screen (used the master CV as the baseline).
3. Palette / font switcher on the review screen — re-runs `assemble.py`
   without re-invoking the brain.
4. Empty states for every list (library, inbox, sources, runs).
5. UX-copy pass with `design:ux-copy` rules: button verbs, plain errors,
   no jargon.

Done when: a full run with `want_cover = true` produces a cover PDF; I can
switch palette on a finished run and see the new PDF without the brain
running; all four lists have honest empty states.

## What I will not do without asking

* Push to a remote.
* Touch `.github/` workflows or CI config beyond adding the smoke test.
* Add a dependency not listed in M0's cross-cutting decisions.
* Auto-format files outside `src/matchbox/` and the new `schemas/`,
  `templates/`, and `tests/` trees.
* Delete anything outside the explicit `git mv` operations in M0.

## Risks I am tracking

1. **Typst not installed locally.** I will detect and surface a clear error,
   but the smoke test will need Typst available — I will flag if the CI env
   lacks it.
2. **`fastembed` model download on first run.** It is ~30 MB. I will gate
   the first call behind a UI message rather than letting it block silently.
3. **ATS endpoints drift.** Workable / SmartRecruiters / Recruitee will be
   probed before they ship. Any that cannot be verified are deferred,
   logged in the milestone notes.
4. **Tailwind via CDN vs build step.** CDN is fine through M6. If the M7
   polish pass needs custom variants, I will switch to a `tailwindcss` CLI
   binary (no Node), not introduce npm.
