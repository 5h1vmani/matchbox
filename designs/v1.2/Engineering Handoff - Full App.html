01 What this is

One app, the whole journey

Matchbox is a single-user, local-first tool. Everything lives in a file on the user's machine; there are no accounts and nothing syncs. The AI assistant runs on a key the user supplies (bring-your-own-key). One machine can hold several people, each with their own file and key.

The app is one continuous journey, expressed as grouped surfaces in a single sidebar:

GroupSurfacesThe job
DiscoverReview · Browse · WatchlistFind real, gettable roles; triage them
TrackToday · ApplicationsWhat needs me now; where everything stands
ApplyReady to apply · Workspace · PeopleGet the whole application out; run interviews; use the network
DecideOffers · InsightsWeigh offers; is this working?
YouProfile · Library · Sources · SettingsKeep my facts trustworthy; configure search and the AI key
Governing principle Honest, calm, and never fabricating. The product earns trust by showing provenance for everything it writes and by stating low confidence plainly. When a rule conflicts with "calm," calm wins. Carry this into every backend response: return reasons and confidence, not just values.

02 Architecture

One React root, a shared global core

The prototype is React 18 + in-browser Babel (no build step), loaded as a sequence of <script type="text/babel"> files that share one global runtime scope. Three formerly-separate apps (tracker, discovery, studio) were unified by collapsing their per-app atom modules into one core.jsx so there are no duplicate global declarations. Port to a bundled React + TypeScript app; the module boundaries below map cleanly to ES modules.

App.html ├─ tokens: colors_and_type.css, mb.css, discovery.css, studio.css, apply.css ├─ data: data.js, discovery-data.js, studio-data.js, apply-data.js (window globals) ├─ ai: ai.js (window.MBAI — provider calls) ├─ core.jsx (Icon, MonoLogo, Badge, all atoms + 3 cross-cutting comps) ├─ aiui.jsx (useStream hook, AISource) ├─ store.jsx, parts.jsx (tracker state + pipeline/quick-action parts) ├─ screens: Today, Tracker, Detail, Review, Browse, WatchlistJD, │ Apply, Tailoring, Intake, Library, Sources, Workspace, │ Offers, StudioInsights, People, Settings, Palette └─ MatchboxApp.jsx (the one shell: sidebar + router + state + chrome) renders into #root
Why one core Each <script type="text/babel"> runs in global scope, so two files declaring function Icon would throw a redeclaration error. Atoms are defined exactly once in core.jsx and exposed on window; screens reference them bare or as window.X. In production this becomes ordinary imports — the constraint disappears, but keep the single-source-of-truth atoms module.

Styling is the Oat design system (see colors_and_type.css tokens) plus the Matchbox brand layer (Hanken Grotesk + JetBrains Mono). Icons are Lucide. Do not introduce new colors or fonts; everything resolves from --* custom properties.

03 Shell, routing & state

MatchboxApp owns everything

A single component holds all app state and renders the sidebar, the active screen, the global chrome (ambient assistant, profile switcher, command palette), and the two drawers (tracker detail + discovery JD). Routing is a route string switched by the sidebar and the palette — replace with your router of choice; routes map 1:1 to URLs.

// state held by MatchboxApp
route        // which surface: 'today' | 'review' | 'apply' | 'people' | …
apps         // tracker applications (useApps store)
roles        // discovery roles + per-role decision
watch        // watchlist companies
profiles, profileId   // local people; the active one
queue        // ambient assistant tasks (see §12)
detail, jd   // open drawer ids (tracker app / discovery role)
paletteOpen, trayOpen, toast

Global chrome present on every screen

  • Ambient assistant chip (bottom-left) — shows queued/working/ready; opens the activity tray.
  • Profile switcher — switches person (and therefore file + AI key). Sets window.__mbProfile.
  • ⌘K command palette — global keyboard entry to any surface or action.
  • Toast — one unified surface; supports a plain message, an undo callback, or an arrival notice with a "Review" action.
Toast contract flash(msg, opts): opts may be a function (treated as undo), or an object { arrival?, go?, undo? }. Arrival toasts (AI results landing) auto-dismiss slower and carry a "Review" button that routes to the result.

04 Data model

Globals & the new TypeScript shapes

The prototype seeds deterministic data on window: STAGES/APPS/PROFILE (tracker), ROLES/WATCH (discovery), STUDIO (profile, facts, tailoring, workspace, offers, insights, assistant, profiles), and APPLY (the new journey). Tracker + discovery shapes are in the prior doc. The new shapes:

// ── the application packet (Apply) ─────────────────────────
interface CoverParagraph { id: string; text: string;
  source: 'your voice' | string;  // 'your voice' or a Library fact id
  note: string; confirmed?: boolean; }

interface ScreeningQuestion { id: string; q: string; answer: string;
  source: string | null;  // 'answer:<id>' | 'profile' | null
  status: 'answered' | 'needs-you'; note?: string; }

interface Packet {
  role: { company; title; mono; location; salary };
  coverLetter: { status: string; paragraphs: CoverParagraph[] };
  questions: ScreeningQuestion[];
}

// ── reusable answers (Library › Answers) ───────────────────
interface Answer { id; question; answer; usedIn: number;
  status: 'verified' | 'thin'; kind?: 'template'; source?; note?; }

// ── people / referrals ─────────────────────────────────────
interface Person { id; name; role; company; mono; relationship;
  strength: 'strong' | 'medium' | 'weak';
  status: 'warm' | 'active' | 'cold';
  lastContact: string; canRefer: boolean; note: string; }
interface WarmPath { company; who; how; peId }   // pipeline co you know someone at

// ── interview loop (Workspace) ─────────────────────────────
interface Round { id; label; who; role; when;
  status: 'done' | 'upcoming' | 'pending'; focus: string;
  debrief: { went: 'well'|'ok'|'poorly'; asked: string[]; notes: string } | null; }

// ── coach + rejection learning (Insights) ──────────────────
interface Momentum { thisWeek: { applied; interviews; followups };
  target; pace: 'good'|'push'|'rest'; headline; message; reframes: string[]; }
interface Patterns { closed: number;
  reasons: { reason; count }[]; insight: string; nudge: string; }
Honesty in the shapes Notice every generated thing carries a source and a note, and screening questions can be needs-you (the system refuses to fabricate). Preserve these fields end to end; they are the product, not decoration.

05 Application packet

The biggest addition: applying, not just a CV

"Ready to apply" (Apply.jsx) is a four-tab packet. Tailoring is now tab one, not the destination.

TabWhat it doesHonesty mechanism
RésuméEmbeds the tailoring diff + coverage (the prior centerpiece)Every change traces to a verified fact; gaps left empty
Cover letterA letter built from facts + the user's genuine motivationParagraphs tagged Your voice vs From your library; "Regenerate" streams live (§11)
QuestionsScreening Q&A pre-filled from the answer library + profileEach answer shows its source; optional questions are needs-you and left blank
SubmitA checklist of packet items + "Mark submitted"Matchbox never submits for the user; submit creates the tracked application

On submit: create an Application in the tracker at stage applied, set a follow-up reminder (default +7d), and surface a warm-path nudge if the user knows someone at that company.

06 Answer library

The Library, extended to Q&A

Library.jsx now has two tabs: Sentences (CV building blocks) and Answers (reusable responses to "Why us?", "Greatest weakness", "Salary expectations" …). Answers carry the same verified/thin status and usage counts. A template answer holds the durable part; the AI tailors the company-specific piece per application (this is what fills the Questions tab in §05). Same honesty model: tailoring may only recombine and personalize what the user already wrote.

07 People & referrals

The network channel

People.jsx. Two parts: a warm-paths banner (companies in the pipeline where the user already knows someone, joined from WarmPath[] against APPS) and the network list with relationship strength + status. "Ask for referral" drafts the message with the AI, calibrated to relationship strength, and streams it into the right panel. The user always sends it themselves; "Mark sent" logs it and sets a nudge.

Backend The warm-path join is the valuable bit: match people's company against active pipeline companies and rank by strength + canRefer. Surface these proactively at the moment of applying (§05 submit).

08 Interview loop

Rounds + debrief, feeding calibration

Workspace.jsx now opens with an interview-loop timeline (Round[]): recruiter → hiring manager → onsite → values, each with status and focus. A completed round with no debrief shows a one-tap capture ("Went well / Hard to read / Rough"). The debrief does triple duty:

  • Sharpens the next round's prep brief (feed asked + notes back into the prep prompt).
  • Feeds honest calibration in Insights (a "rough" pattern at a stage is signal).
  • Surfaces Library gaps (a question they fumbled → a thin answer to strengthen).

Below the loop sit the prep brief and the timed follow-up / thank-you drafts (unchanged from the prior workspace).

09 Momentum & learning

The emotional job, honestly

Insights (StudioInsights.jsx) gained two honest, non-gamified layers:

  • Coach / momentum — weekly pace (applied / interviews / follow-ups), a pace read ("healthy" / "push" / "rest"), and truthful reframes drawn from real funnel data ("you reached onsite in 3 of 5 — that's working"). It tells people to rest when the pipeline is healthy. No streaks, no nagging.
  • Learning from rejection — closed roles are categorized (Patterns.reasons) and turned into one insight + one nudge ("your misses were timing and comp, not craft → work your network"). Capture the reason when a role is closed; this is the feedback loop the whole category drops.

10 Command palette

⌘K, the daily-driver spine

Palette.jsx. Cmd/Ctrl+K toggles an overlay over commands (built in the shell): a "Go to" group (every surface) and an "Actions" group (tailor a CV, add an application, ask for a referral, open assistant activity). Fuzzy filter, arrow-key navigation, Enter/click to run. Extend commands as you add surfaces — it's a flat array of { id, group, label, icon, hint, run }.

11 BYOK client

ai.js — one interface, real or simulated

All generation goes through window.MBAI.stream(). It reads the active person's key from localStorage (mb_key_<profile>, set in Settings) and tries, in order: the user's provider → window.claude → a calm local simulation. It always resolves, so the UI behaves identically with or without a key.

interface MBAI {
  config(): { provider: 'anthropic'|'openai'|'local'; key: string; on: boolean };
  hasKey(): boolean;
  stream(opts: {
    system?: string;
    prompt: string;
    fallback?: string;        // streamed out if no live provider
    onToken?: (full: string) => void;  // cumulative text, every chunk
    signal?: AbortSignal;
  }): Promise<{ text: string; source: 'byok'|'claude'|'demo' }>;
}

The client implements proper SSE streaming for Anthropic (/v1/messages, anthropic-dangerous-direct-browser-access) and OpenAI (/v1/chat/completions). React consumes it via the useStream() hook (aiui.jsx): { text, busy, source, run, stop }. The AISource badge tells the user where output came from — "Live · your key" / "Live · Claude" / "Demo — add a key for live AI" — which is part of the honesty contract.

CORS / production Browser-direct provider calls can be blocked by CORS in some embeds; the client then falls back to demo output. For production, route MBAI.stream through your own backend proxy that holds nothing (the key still comes from the user/device) but terminates TLS and SSE server-side. The interface stays the same; only the two fetch URLs change to your endpoint. Keys must never be logged or persisted server-side — local-first is the promise.

12 Ambient async state

Work is visible, never blocking

Every live generation registers a task so the assistant feels like a colleague, not a spinner. The shell exposes window.MBQueue; useStream calls it around each run:

interface MBQueue {
  start(label: string, kind?: string): string;  // pushes a 'working' task, returns id
  done(id: string): void;                    // flips it to 'ready'
}

Tasks render in the sidebar chip (a breathing dot, "Assistant is working") and the activity tray (working / queued / ready). When a task completes off-screen, an arrival toast ("Your tailored CV for Vercel is ready") offers a "Review" jump. States: queued → working → done. No modal loaders anywhere; the streaming text itself is the progress.

13 Prompts & the honesty contract

The system prompt is a guardrail

Every generation ships a system prompt that forbids fabrication. The pattern, applied across tailoring, cover letters, answers, prep, and referral drafts:

"You are a job-seeker's honest writing assistant. Write ONLY from facts
the user has verified. Never invent achievements, numbers, employers, or
claims. If a requirement isn't supported by their library, leave it out and
say so. Warm, plain, no clichés, no hype."

Generation requests pass the relevant verified facts as the only source material. The backend (or the model) must return, alongside prose: which facts were used (provenance), and which requirements could not be covered (the honest gaps). The UI already has slots for both — keep them populated. Anything the model can't ground, the UI shows as empty, never filled.

This is the moat The diff-and-coverage view, the "your voice" tags, the confidence dashes, and the "demo vs live" badge all exist to make "the machine wrote this" feel safe. Don't optimize them away for polish — they are why a stressed user trusts the output enough to send it.

14 Backend contract

What engineering needs to provide

The front end is complete and mock-driven. To make it real, the backend supplies these capabilities (ignoring exact transport — REST, local IPC, or in-process is your call):

Role search & parseDiscover, SourcesRoles + JD text from boards or a pasted link/description
Fit + eligibility scoringDiscover, TailoringLevel + a one-line reason; eligibility status + the blocker

Two cross-cutting rules for every response: (1) include a reason and a confidence/source, never a bare value; (2) it is correct to return "we don't know" or "left empty" — the UI is built to show that as a kindness.

15 Persistence & profiles

Local-first, multi-person, no accounts

  • One file per person. All of a person's data (facts, roles, applications, answers, people, settings) lives in a single local file. The "Saved locally" chip and the file names in the switcher imply this; define the format (SQLite, a JSON document, or a CRDT for future multi-device).
  • Profile switch = swap the open file. The switcher sets window.__mbProfile; the AI key is namespaced per profile (mb_key_<profile>). Keep keys out of the data file if the file is ever shared; the prototype keeps them in localStorage.
  • Dates. The prototype uses relative daysAgo integers for determinism. Production stores ISO timestamps and derives the offsets.
  • Nothing leaves the device except the explicit generation request the user sends to their chosen provider with their own key. Make that boundary visible and auditable.

17 Open questions

Decisions for product/eng before build

  1. AI transport. Browser-direct (simplest, CORS-limited) vs a local backend proxy (recommended). Confirm which, and the exact endpoint for MBAI.stream.
  2. Local file format. SQLite vs JSON doc vs CRDT. Decides export, portability, and any future multi-device story.
  3. Scoring source. Where fit/eligibility/coverage/salary come from, and how each reason + confidence is produced. The UI requires them; define the service.
  4. Submission integrations. "Mark submitted" is manual today. Any board/portal autofill? ATS exports?
  5. Reminders & notifications. In-app only, or OS/calendar (interviews, follow-ups, offer deadlines)?
  6. First-run. The onboarding (drop a CV → watch facts appear) and zero-data empty states aren't built. Highest-leverage next design.
  7. Model + cost. Default models per provider, token budgets, and how to keep BYOK usage transparent to the user.
  8. Accessibility pass. Drawers → role="dialog" + focus trap; hover-only actions also on focus; palette already keyboard-first. Schedule the full audit.
Reference The running prototype is the source of truth for behavior and pixels: open App.html, press ⌘K, set a key in Settings, and hit "Regenerate" on a cover letter to watch the live streaming + ambient state.
