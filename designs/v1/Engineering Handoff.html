01 Overview & goals

One screen, one job: "what do I do now, and where does everything stand?"

The user is a single, stressed job-seeker tracking every role they are pursuing — not a recruiter managing a funnel. The product is local-first and single-user. Every design decision serves four priorities, in order:

  1. Tell them what to act on today — follow-ups due, interviews coming up, drafts ready to send, applications gone stale.
  2. Show where everything stands at a glance — the pipeline.
  3. Stay calm and honest — never anxiety-inducing, never gamified. Stale and closed applications are shown truthfully, without nagging.
  4. Let them act fast inline — advance a status, set a reminder, log a response, jot a note — without leaving the page.
Design principle When a rule and "calm" conflict, calm wins. No red badges on counts, no streaks, no "you're behind" language. Overdue items are stated plainly and grouped, not amplified.

02 Screens

Three surfaces plus a detail drawer

Navigation is a fixed left sidebar. The default landing is Today. The detail drawer slides in over whichever screen is active, so the user never loses their place.

Today / Focus home default route

Answers priority #1 first. Sections render top-to-bottom in urgency order:

  • Greeting + glance row — time-of-day greeting, today's date, and four read-only counters: to do today, interviews this week, active, offers (offers only shown when > 0).
  • Today list — today's and overdue tasks as a checklist. Each row: complete-tick, action line, due pill, a "Draft" badge when a draft is ready, and on hover a Snooze button + quick-action menu. Capped to 7 visible with a "Show N more" expander so a busy day never reads as a wall.
  • Upcoming interviews — next interviews (0–7 days), each with a date/time and a Prep button.
  • Drafts ready to send — follow-ups / thank-yous already drafted, each with a one-click Send.
  • Going cold — collapsible, dimmed group of stale applications. Collapsed by default. Per row: Follow up (sets a reminder) or Let go (logs no-response). This is the honest-but-not-nagging treatment of stale work.

Applications tracker route: /applications

  • Pipeline panel — the at-a-glance view of priority #2. Renders as a proportional segmented bar (ledger) or a funnel (focus). Each stage is a filter; clicking toggles it.
  • Toolbar — search (company / role / location) and a sort cycle (Next action → Last updated → Company).
  • List view — rows grouped by stage. Columns: company + role, next action + due (+ draft badge), salary + last-updated, star + quick-action menu. Stale rows are dimmed to ~60% opacity and brighten on hover.
  • Board view — one calm column per stage with compact cards. Same data, horizontal scroll.

Application detail drawer overlay

Right-docked, 560px, slides in over the active screen. This is where priority #4 lives — everything can be done here:

  • Header — monogram, company, role, and tag chips (stage, location, salary).
  • Stage stepper — honest progress across the pipeline; dims entirely when closed.
  • Next-action card — the current action with a primary CTA (Mark sent / Mark applied / Accept offer …) and Snooze. Falls back to a "Waiting to hear back" or "Closed" state.
  • Quick actions grid — Advance stage · Log a reply · Set a reminder · Mark closed.
  • Details — applied date, last update, salary, source.
  • Notes — list + inline composer (⌘/Ctrl+Enter to save).
  • People — contacts with an email affordance.
  • History — reverse-chronological timeline of events.

Insights route: /insights

A light, supporting read on priority #2 ("is my search healthy?"). Honest stats only — active count, response rate, interviewing, offers — plus the pipeline viz and a calm note about what's gone cold. No streaks, no vanity metrics.

03 Two directions

A single prop swaps the whole feel

The prototype ships two approved directions behind one value, direction: 'ledger' | 'focus'. It threads from app state into two components; everything else is shared. Pick one for v1 (recommend shipping ledger as default, focus as an optional preference) or keep both as a setting.

AspectCalm ledgerFocus & flow

Two presentational prefs ride alongside: density: 'regular' | 'compact' (set on the shell as data-density) and accent: 'taupe' | 'forest' | 'slate' (swaps the --oat-* custom properties at runtime).

05 Data model

TypeScript shapes

The prototype generates ~45 deterministic records in data.js. Production types below. Dates in the prototype are stored as relative daysAgo integers for determinism — production should store ISO timestamps and derive the day offsets.

type StageId =
  | 'saved' | 'applied' | 'phone' | 'onsite' | 'offer' | 'rejected';

interface Stage {
  id: StageId;
  label: string;   // "Phone screen"
  short: string;   // "Screen" — used on board/stepper
  tone:  string;   // hex; the stage's dot/bar color
}

type ActionKind =
  | 'apply' | 'followup' | 'interview' | 'prep'
  | 'thanks' | 'offer'  | 'wait';

interface NextAction {
  kind:  ActionKind;
  label: string;          // "Send follow-up"
  due:   number | null;   // days from today: <0 overdue, 0 today, null = no date
  time?: string;          // "13:30" for interviews
}

interface TimelineEvent {
  daysAgo: number;
  kind: 'saved'|'applied'|'reply'|'screen'|'onsite'|'offer'|'rejected'|'note'|'advanced'|'followup';
  text: string;
}

interface Contact { name: string; role: string; initials: string; }
interface Note    { daysAgo: number; text: string; }

interface Application {
  id: string;
  company: string;
  role: string;
  location: string;
  salary: string;          // display string, e.g. "$150–185k"
  source: string;          // "Referral · Dana"
  stage: StageId;
  appliedDaysAgo: number | null;  // null while still 'saved'
  updatedDaysAgo: number;         // drives staleness + "last update"
  nextAction: NextAction | null;
  hasDraft: boolean;             // a follow-up / thank-you is written & ready
  events: TimelineEvent[];
  contacts: Contact[];
  notes: Note[];
  starred: boolean;
  mono: { bg: string; fg: string };  // monogram colors
  stale: boolean;               // DERIVED — see §07, recompute, don't persist
}

06 State & actions

One store, a flat set of inline actions

The prototype centralizes mutations in a useApps() hook (see store.jsx). Every action updates the record and writes a timeline event, then surfaces a toast. Port this to your store of choice (reducer, Zustand, signals) — the contract is what matters:

interface TrackerActions {
  advanceStage(id: string): void;                 // → next stage in FLOW
  setStage(id: string, stage: StageId): void;     // explicit move (incl. close/reopen)
  snooze(id: string, days?: number): void;        // default +2; pushes the action's due out
  remind(id: string, days: number): void;         // set/replace the due (0 = today)
  markDone(id: string): void;                     // complete current action, clear draft
  logResponse(id: string, type: 'reply'|'rejected'|'ghosted'): void;
  addNote(id: string, text: string): void;
  toggleStar(id: string): void;
}
ActionEffectTimeline event
advanceStageMove to next stage in FLOW; set a sensible default next action; updatedDaysAgo→0; stale→false"Moved to …"
markDoneLogs the current action done; clears hasDraft; an apply moves saved→applied; a sent follow-up becomes "Waiting to hear back""Follow-up sent" / "Applied" …
logResponse('reply')applied→phone; otherwise keeps stage but clears stale"Heard back — positive"
logResponse('rejected')stage→rejected; clears next action"No longer moving forward"
logResponse('ghosted')Clears next action; marks stale (no shame, just true)"Marked as no response"
snooze / remindShifts / sets the action's due"Reminder set …" (remind only)

07 Business rules

The logic that makes it feel right

Stage flow

Saved→ Applied→ Phone screen→ Onsite→ Offer · Closed (terminal)

FLOW = ['saved','applied','phone','onsite','offer']. rejected ("Closed") is off-flow and terminal but reopenable via setStage.

Due buckets (drives the due pill + sort)

ConditionLabelTone
due < 0"Nd overdue"danger
due === 0"Today"warning
due === 1"Tomorrow"muted
2 ≤ due ≤ 6"in N days"muted
due > 6"in N days"faint

Staleness ("Going cold") — derived, never persisted

isStale(a) =
  a.stage in {'applied','phone','onsite'}        // only active stages
  && !(a.nextAction && a.nextAction.due <= 3)  // no imminent action
  && a.updatedDaysAgo >= 11;                  // untouched ~1.5+ weeks

Saved, offered, and closed applications are never stale. Recompute on read.

"Today" selection & ranking

An application appears in Today when nextAction.due !== null && due <= 0 && stage !== 'rejected' && kind !== 'wait'. Sort by kind-rank, then by due (most overdue first):

  • Rank 0 — interview, offer (time-sensitive, high stakes)
  • Rank 1 — thanks, prep
  • Rank 2 — apply, followup

The focus hero shows today[0], falling back to the nearest upcoming interview when Today is empty. Upcoming interviews = kind==='interview' && 0 ≤ due ≤ 7. Drafts = hasDraft && stage !== 'rejected'.

Type

  • UI: Hanken Grotesk (400/500/600)
  • Data/numerals: JetBrains Mono, tnum on — salary, dates, counts
  • Base 16px / 1.5; UI text 14px; badges 12px
  • Headings 600, tight; text-wrap: balance

Shape & depth

  • Radius 6px default; pill for badges/avatars; 10–12px on cards
  • Structure from 1px borders, not shadows
  • Shadows barely-there: 0 1px 2px rgba(0,0,0,.05)
  • No gradients (the focus hero is the one soft wash); no glassmorphism

10 Interaction & motion

  • Transition: .12s cubic-bezier(.4,0,.2,1) on background, color, opacity, transform. Fast and subtle.
  • Hover: faint --muted fill on rows/buttons; board cards lift 1px. Row quick-actions fade in on hover (and must be reachable via keyboard focus, not hover-only — see §12).
  • Press: ~0.5px translate nudge. Never bouncy.
  • Drawer: slides in ~220ms from translateX(24px); scrim fades. Close on Esc, scrim click, or the X.
  • Complete a task: tick fills, line strikes through for ~280ms, then the row leaves the list (commits markDone). Gentle, not celebratory.
  • Toast: single, bottom-center, ~2.4s auto-dismiss. Consider an Undo affordance on destructive logs (close / let go).
  • Respect prefers-reduced-motion; no decorative looping animation anywhere.

12 Accessibility

  • Real semantic elements throughout — <button>, <nav>, <input>. The drawer should be a proper role="dialog" with focus trap + restore, labelled by the company name.
  • Visible focus ring in the brand brown (--ring), 2px, ~3px offset halo. Keep it on every interactive element.
  • Hover-revealed actions must also appear on keyboard focus (the row "…" menu, snooze, star) — don't gate them on :hover alone.
  • Due state and stage must not rely on color alone — they always carry a text label.
  • Quick-action menu: arrow-key navigation, Esc to close, focus returns to the trigger.
  • Hit targets ≥ 40px where practical; the complete-tick and icon buttons currently sit at ~21–32px — pad their tap area.
  • Honor prefers-reduced-motion and prefers-color-scheme (Oat tokens already define dark via light-dark(); dark wasn't designed this pass — see §15).

13 Voice & copy

Calm, plain, on your side

  • Sentence case everywhere. Never Title Case, never ALL CAPS (except tiny letter-spaced eyebrows).
  • No em dashes, no double hyphens. Use a period, comma, or parentheses.
  • Second person, present tense. "3 follow-ups due today", not "You may wish to address…".
  • Verbs on buttons: "Log application", "Snooze", "Mark applied", "Add note".
  • Lead with the number when it helps you act. No emoji. The warmth is in the words and color.
  • Empty/quiet states are kind: "Nothing needs you today. Go for a walk." not "No records found."

❌ "You have not updated this application in a while, perhaps follow up."  →  ✅ "No update in 12 days. Send a follow-up?"

15 Open questions

Decisions for product before/while building

  1. One direction or a preference? Ship ledger as default; is focus a setting, or do we drop it for v1?
  2. Persistence model. Local-first is core — IndexedDB? A single local file (the "Saved locally" chip implies a file)? Define the source of truth and the date format.
  3. Reminders & notifications. Are due dates purely in-app, or do they fire OS/browser notifications? That changes the "remind me" UX.
  4. Add / import applications. Manual form, paste-a-URL parse, or email forwarding? Out of scope here but it's the top of the funnel.
  5. Board interactivity. Drag-to-reorder / drag-between-stages was not built — wanted for v1?
  6. Dark mode. Tokens support it; the screens weren't designed in dark this pass. Schedule a dark pass if it ships.
  7. Drafts. The prototype models hasDraft as a flag. Where does draft content live, and does "Send" actually send, or mark-as-sent?
