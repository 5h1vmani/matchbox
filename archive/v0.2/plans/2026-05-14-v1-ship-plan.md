# v1 ship plan — the Apple / Anthropic way

**Date:** 2026-05-14
**Companion to:** [docs/audits/2026-05-14-sota-audit-and-roadmap.md](../audits/2026-05-14-sota-audit-and-roadmap.md)
**Owner:** Shiva (solo maintainer)
**Target ship date:** 4 weeks of focused effort from start.

This plan turns the audit into doing. It is opinionated. It locks decisions where the audit left them open. It sequences work so that the highest-leverage things ship first.

The Apple half of this plan is the discipline: ship five things flawlessly, not twenty things half-built. The Anthropic half is the posture: be honest about limits, be transparent about cost, default to safe.

---

## 1. Mission for v1

A technical jobseeker can clone the repo, set up a profile in 15 minutes without editing YAML, and have a working triage queue with real match scores, tailored materials, and tracked outcomes. The tool earns trust in the first hour by being honest about what it does and does not score, by never spending money without confirmation, and by never sending data off the machine unless the user asked it to.

That sentence is the only acceptance test that matters. Every v1 decision serves it.

---

## 2. The five promises to the v1 user

These are the things v1 commits to. They are written from the user's point of view.

1. **You will not edit YAML on your first day.** The web UI handles the parts that matter for onboarding (who you are, what you want, what you do not want).
2. **You will see scores you can trust.** Every dimension shown is a real measurement. There are no padded placeholders.
3. **You will know what every action costs before you confirm it.** And the tool will never silently spend.
4. **Your data stays on your machine** unless you choose otherwise. Telemetry is opt-in, plain-English, and you can see exactly what would be sent before sending.
5. **You can start using it in 60 seconds** from a single link, no install.

---

## 3. The anti-promises

These are the things v1 will not do. Anti-promises matter more than promises because they keep scope honest.

1. v1 will not be multi-user. There is no auth, no team mode, no shared database.
2. v1 will not score cultural fit or compensation as a number. Those dimensions are dropped from the rubric, not faked.
3. v1 will not have a mobile layout, a chat sidebar, an LLM-powered job-board feed, or a one-click apply button.
4. v1 will not include interview prep, dream-job gap analysis, or skill-coaching. Those are v2.
5. v1 will not add ATS types beyond what is named in the README. If Workable cannot be implemented cleanly, the README is corrected instead of the code.
6. v1 will not auto-detect responses from your email. The lightweight paste-email path ships; full IMAP integration is v2.

The product team that ships v1 says no to feature requests by citing this list, not by negotiating.

## 4. Decisions locked by this plan

The audit left seven open questions. The plan answers them now so engineering does not stall on debate.

### 4.1 Placeholder scores: drop them, re-weight to four dimensions

**Decision.** Remove `comp_score` and `cultural_score` from the scoring rubric for v1. The four remaining dimensions (`cv_match`, `company_mission_fit`, `role_mission_fit`, `red_flags`) become the score. The profile schema keeps backward compatibility by ignoring the two old weight fields with a one-line migration note.

**The Apple reasoning.** Shipping a score that is partly a fixed 3.0 is shipping a lie. Apple does not ship lies. They cut.

**The Anthropic reasoning.** Two things the score breakdown UI must show explicitly: which dimensions are scored, and which dimensions are deliberately not. A short link from the UI to a doc page titled "Why no cultural-fit score" explains the choice in three sentences. Users trust tools that explain their own gaps.

**Adjacent UX change.** When a job listing states a salary range explicitly, the detail panel shows it as a non-scoring badge labelled "Stated comp: $X–$Y". When the JD does not state comp, the badge is absent — there is no inference, no scraping, no Levels.fyi lookup. This keeps the comp surface visible without faking a number.

### 4.2 Hosted demo: GitHub Codespaces + asciinema, no paid hosting

**Decision.** No Fly.io, no Railway, no Render. Two unconventional pieces tick the box for zero recurring cost.

* A `.devcontainer/devcontainer.json` configures the repo so the README's "Open in Codespaces" button launches a real running Matchbox in the user's browser, seeded with the demo profile, no API key required. Cost to us is zero. Cost to the user is zero up to GitHub's free Codespaces hours per month.
* An asciinema recording of the CLI walkthrough plays inline in the README. Asciinema embeds are SVG, look live, and are hosted free.
* Three static screenshots and one 25-second GIF (recorded with a free macOS tool) round out the visual story.

**Effort:** 1.5 days for the three pieces combined.

**Why this is better than a real hosted instance.** A hosted demo means we operate infrastructure forever. We worry about API key abuse, cost runaway, uptime, and seeded-data resets. Codespaces shifts every one of those concerns to the user's GitHub account where they belong. The result feels just as immediate.

### 4.3 Naming: pause public launch until rename decision is made

**Decision.** Treat "Matchbox" as a working name, not a launch name, until the conflicts are resolved. The audit's web check found:

* PyPI package `matchbox` is taken (unrelated project; blocks `pip install matchbox`).
* Three direct collisions in the HR/recruiting space: theMatchBox.ai, Matchbox Tech (Japan), MatchBox Consulting Group.
* A trademark filing exists (serial 85769912); class and current status unverified.

**Action.** Before public launch, do two things. First, run a USPTO TESS search and a basic EU/UK trademark search to see if the 2012 filing is live and what class it covers. Second, decide whether to rename or to live with the conflicts. The renaming question is out of scope for the engineering plan but is a prerequisite for the public launch step at the end.

**Working assumption for the plan.** Engineering work continues under the name Matchbox. Any name change is a global rename late in the cycle — cheap if done before launch, expensive after.

### 4.4 Telemetry: opt-in default off, PostHog Cloud, transparent by design

**Decision.** Telemetry is built but is **off by default**. A user enables it with `matchbox telemetry on` after seeing what would be sent.

**What gets collected.**
* *Install ping (one-time):* anonymous UUID, OS family, Python version, Matchbox version, install date. No company names, no job titles, no profile content, no API usage details.
* *Weekly aggregate (optional, separately opt-in):* jobs scanned, jobs tailored, total spend in USD, average score. All scalar; no text fields.
* *Crash reporter (separately opt-in):* scrubbed traceback, Matchbox version. Paths and identifiers stripped. No JD text, no profile content.

**Where it goes.**
* **PostHog Cloud** free tier. PostHog is open-source itself, which is on-brand for this project: an OSS project uses another OSS project for its telemetry, and the user can read PostHog's source to verify what happens server-side.
* Free tier covers a million events per month, far beyond any realistic v1 volume.
* If volume ever justifies it, self-host PostHog — its code is on GitHub and the deploy path is well documented.

**How the pipeline works.**

1. The Python client builds the payload, calls PostHog with a five-second timeout, and returns whether or not the call succeeded. Failures are silent.
2. PostHog stores events; we read them via PostHog's dashboard.
3. A public dashboard on the project's docs site shows aggregate counts only ("8,432 installs across 47 countries"). No individual data.
4. Every quarter, a transparency note in the changelog summarises what telemetry told us. Builds trust.

**The commands the user controls.**
* `matchbox telemetry show` prints the exact JSON that would be sent. No surprises.
* `matchbox telemetry on` enables, after showing the payload one more time and asking for confirmation.
* `matchbox telemetry off` disables. The default.
* `matchbox telemetry status` shows current state.

**What else we can do in this space** beyond the basic install ping.
* **Public aggregate dashboard.** The single best trust signal an OSS telemetry program can offer. Users see what we see. Drives more opt-ins because it is clearly not surveillance.
* **30-day post-install survey** (separately opt-in). One yes/no question and an optional free-text comment. Gives real qualitative signal without scraping forums.
* **Quarterly transparency post.** What telemetry told us. What we learned. What we changed because of it.
* **A `telemetry-policy.md` doc** linked from every consent prompt. Plain English. No legalese.

The Anthropic posture here is the entire game. Tell the user everything before asking. Let them verify. Then earn trust over time.

### 4.5 Contributor model: open contribution, no formal collaborator circle

**Decision.** No pre-anointed maintainer team. Anyone who contributes thoughtfully and consistently can earn merge rights over time. This is documented in `GOVERNANCE.md` and `CONTRIBUTING.md`.

**Practical guardrails:**
* One maintainer (Shiva) has merge rights at v1.0.
* A contributor who has shipped two high-quality PRs and engaged respectfully in two discussions can be offered triage permissions (label issues, close duplicates, but not merge).
* Merge rights remain a slower trust-earning path: at least one published architectural-discussion thread plus four merged PRs, with no maintainer hesitation about the work.
* Trust is revocable for cause; documented in `GOVERNANCE.md`.

### 4.6 Sources seed license: noted, no special licensing required

**Decision.** The 20-org default seed in `sources.yaml` lists publicly known ATS slugs. No licensing concern. A one-line note in `CONTRIBUTING.md` reminds contributors not to add anything that is not public information.

### 4.7 Issues vs Discussions: clear split, enforced through templates

**Decision.** Issues are for bugs and concrete tasks the maintainer has accepted. Discussions are for everything else: feature requests, questions, "is this for me," idea threads. The issue templates explicitly route feature requests to Discussions. The discussion templates explicitly route bug reports to Issues. This is enforced in the templates themselves so users do not have to learn the convention.

## 5. Sequenced work blocks

Each block has a single outcome, a single reason, an effort estimate, and a definition of done. They run in order because each unblocks the next.

### Block A — Honesty (week 1)

Fix the things that would shame the project on its first day in public.

**Outcomes.**
* The four-dimension rubric is live. Two stub dimensions are removed. The score breakdown UI explains the choice.
* README claims match code. The Workable claim is either implemented or removed.
* The audit and this plan are linked from `ROADMAP.md` so reviewers can read the thinking that produced the v1 scope.

**Why first.** A new user judges a tool in the first ninety seconds. The fastest way to lose them is for them to discover something dishonest. Fix the dishonesty before fixing anything else.

**Effort.** Five to six working days.

**Definition of done.**
* The rubric in code has four dimensions.
* The score-breakdown view in the web UI displays only those four and links to "Why no cultural-fit score."
* The README reflects the actual ATS list.
* `ROADMAP.md` exists at the repo root with the v1/v2 split.

### Block B — On-ramp (week 2)

Turn the demo into the first contact a stranger will fall in love with.

**Outcomes.**
* The web UI has a profile editor covering candidate, targets, and filters. Atomic save reuses the existing temp-and-replace infrastructure.
* The README opens with three screenshots and one short GIF.
* A GitHub Codespaces devcontainer is wired up. The README has an "Open in Codespaces" button at the top.
* An asciinema recording of the CLI walkthrough is embedded under the hosted demo button.
* A "What Matchbox is not" section sits above the install instructions. It is short and direct.
* A "vs Teal / Huntr / Simplify" paragraph appears in the README, written without throwing shade.

**Why second.** Block A makes the product honest. Block B makes the product approachable. Approachability without honesty is fraud; honesty without approachability is a dotfile repo with strong opinions.

**Effort.** Five working days.

**Definition of done.**
* A user lands on the README, clicks "Open in Codespaces," and has a running web UI with the seeded demo profile within three minutes. They do not see YAML.
* A different user who installs locally can create their profile entirely through the web UI without touching `profile.yaml`.

### Block C — Reach (mid week 3)

Make the tool genuinely useful for a user whose target companies are not on the hardcoded list.

**Outcomes.**
* `KNOWN_SOURCES` becomes `people/{name}/sources.yaml`, seeded with the current twenty.
* The web UI supports adding and removing orgs by ATS type with a one-line form.
* A "+ Add job by URL" affordance on the Inbox accepts either a job-listing URL or pasted JD text. The server fetches the URL, extracts plain text, scores the job, and drops it into the queue.
* One simple cron recipe each for macOS launchd and Linux systemd is added to `docs/operator-runbook.md`. The CLI does not gain a scheduler.

**Why third.** A new user with their own target list and their own job-finding habits can now use the tool for their real situation, not just the demo's.

**Effort.** Four working days.

**Definition of done.**
* A user adds three companies of their choosing via the web UI and gets jobs from them on the next scan.
* A user pastes a LinkedIn job URL and sees a scored entry on their Inbox within fifteen seconds.

### Block D — Trust (late week 3)

Build the telemetry layer described in Section 4.4. This block is small but high-stakes because the user-facing copy is the product.

**Outcomes.**
* Three new CLI commands work: `telemetry show`, `telemetry on`, `telemetry off`, `telemetry status`.
* A PostHog Cloud project receives anonymous install pings only from users who have explicitly opted in.
* The `docs/telemetry-policy.md` page exists in the docs site and is linked from every consent prompt.
* A scaffold for the weekly-aggregate event and the crash reporter is implemented and disabled. They turn on in v1.1 once the install-ping baseline is healthy.

**Why fourth.** Telemetry without on-ramp is useless because nobody is installing. Telemetry without honesty is creepy. After Blocks A through C, telemetry is the right next step.

**Effort.** Three working days.

**Definition of done.**
* A user can run `matchbox telemetry show` and read the exact JSON that would be sent.
* The default install does not send anything to anybody.
* Opt-in is a two-step flow: show payload, then confirm.

### Block E — Polish (week 4)

The last-mile cluster. Each item is independently shippable; ship the ones that are cheap, skip what is not.

**Outcomes (in priority order).**
* Cross-board dedup using normalized company and role.
* ETag and last-modified caching for ATS probers; a 250 ms polite delay between board requests.
* Snooze action on a job row (push back N days).
* Saved filter presets per profile.
* Cost-this-week widget on the Insights page.
* Hard minimum clamp on `MATCHBOX_COST_CONFIRM_USD` (e.g. cannot be set below ten cents).
* Paste-email response classifier (regex first, LLM fallback only at low confidence).

**Why last.** Each of these is a small joy. They are also the easiest to defer if Blocks A through D run long. Treat this block as compressible.

**Effort.** Four to five working days depending on how many of the seven items survive.

**Definition of done.**
* At least four of the seven items ship.
* The two safety items (`MATCHBOX_COST_CONFIRM_USD` clamp, dedup) ship regardless.

### Block F — Launch (end of week 4)

Take v1 to the public.

**Outcomes.**
* `CHANGELOG.md` is updated with the v1.0 entry.
* Issue and PR templates land in `.github/`.
* Three seeded discussions are posted: "How does Matchbox score?", "How do I run this on a remote machine?", "Show us your sources.yaml."
* `GOVERNANCE.md` exists at the repo root and matches Section 4.5 of this plan.
* A short launch post explains the wedge and links to the demo. It avoids comparison shopping.
* The project posts itself to one or two surfaces that match the user, not to everywhere. Hacker News once. r/cscareerquestions once. One technical newsletter that fits.

**Effort.** Three working days, plus the launch day itself.

**Definition of done.**
* v1.0 is tagged.
* The public can install it.
* The maintainer is rested and present for the first 72 hours of feedback.

## 6. Acceptance criteria for v1.0

These are the gates v1.0 must pass before tagging.

1. **The mission sentence in Section 1 is true** when a non-maintainer tester runs through the flow cold.
2. **No placeholder data in the product surface.** Every number the user sees comes from a real source.
3. **No surprise spend.** The cost-confirmation flow cannot be bypassed by env vars below a sane floor.
4. **Telemetry default is off.** A fresh install sends nothing.
5. **The README opens with a working demo path** that does not require installation.
6. **The `docs/audits/` and `docs/plan/` documents that produced this plan are linked from `ROADMAP.md`.** Reviewers can read the thinking.
7. **Tests pass at strict.** Ruff, mypy strict, pytest green in CI.
8. **A new user can complete onboarding without opening a YAML file.** This is the single hardest acceptance test. Test it with one real person who is not the maintainer before tagging.

## 8. Open decisions that still need an answer

The plan locked seven decisions. These four remain.

1. **Rename or live with the conflicts.** Section 4.3 explained the conflicts; the decision itself is yours. A rename is cheapest before launch.
2. **Whether the weekly-aggregate telemetry event and the crash reporter ship in v1.0 or wait for v1.1.** The current plan ships only the install ping in v1.0; the other two are scaffolded but disabled.
3. **Whether Codespaces hours are enough as the only demo path,** or whether a tiny Cloudflare Worker fallback is worth half a day to render a static "demo snapshot" page (for users who do not have a GitHub account).
4. **Whether to enable GitHub Sponsors / OpenCollective at v1.0** or wait until v1.x. Recommend wait — opening for sponsorship before the project has a track record signals weakness, not value.

## 9. Risks and mitigations

Three risks are large enough to call out.

**Risk: the YAML wall is not closed in v1.** If Block B slips, the on-ramp fails and the launch loses energy.
*Mitigation.* Block B has the largest slack budget on purpose. If anything has to be cut, cut the GIF before cutting the profile editor.

**Risk: the placeholder-score removal breaks existing seed profiles.** A weight field that no longer maps to a dimension may cause confusing behavior.
*Mitigation.* The schema migration logs a clear warning for unknown weight keys, normalizes the remaining weights to sum to one, and prints what it did.

**Risk: telemetry blows up trust.** A single rumor of "Matchbox phones home by default" on a forum can sink the launch.
*Mitigation.* The default is off, the consent flow shows the payload twice before any send, the policy doc is linked from everywhere, and the public aggregate dashboard goes live with v1.0 so users can verify that what we see is what we say we see.

A fourth, smaller risk: maintainer burnout in the first 72 hours after launch. The release is timed so the maintainer is not also juggling a separate launch elsewhere. If a release week is also a busy week somewhere else, push the launch by a week.

We will ship v1 when a stranger can install Matchbox, build a profile without YAML, see four honest scores on their first ten jobs, tailor one of them, and know exactly what it cost — all without us collecting anything they did not consent to.

When the team disagrees about a v1 feature, read that sentence first.
